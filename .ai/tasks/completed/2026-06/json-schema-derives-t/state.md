## Mission

Ship the schema-derives-T capability as a new packlet in `@fgv/ts-json-base`: 7 factories + `optional` modifier + `Static<S>` + `fromJson` + supporting types, for the LLM-tool JSON Schema subset. Schema values ARE the validators (`ISchemaValidator<T>` extends `Validator<T>`). Strictly additive; no existing surface change anywhere in the monorepo.

## Decisions log

| Decision | Resolution | Rationale |
|---|---|---|
| **(In-flight #1) Packlet name** | `json-schema-builder`, exported as the `JsonSchema` namespace from `@fgv/ts-json-base` | Recommended default. Directory name describes the capability (typed schema *building*); the consumer-facing namespace `JsonSchema` matches the name used in the spike's drift-test sketch (`research.md` §3.3) and the feasibility doc. |
| **(In-flight #2) Strict numeric mode** | Default **strict** (reject numeric strings) on `number()` / `integer()`; opt out via `{ strict: false }` | LLM tools emit JSON numbers as numbers, so strict is the right default and avoids the `Converters.number` numeric-string coercion (`'42'`→`42`) that `research.md` §1.5 flagged as surprising. Strict is composed from existing `Converters.isA(...)`; non-strict reuses `Converters.number` (+ an integer `withConstraint`). `fromJson` produces strict numeric converters (no wire keyword to express the opt-out). |
| **(In-flight #3) `additionalProperties` default** | Default **`false`** (reject unknown fields → `Converters.strictObject`); opt in via `{ additionalProperties: true }` (→ `Converters.object`) | LLM-tool convention. `toJson` emits `additionalProperties: false` only when false (true is the JSON Schema default and is omitted). `fromJson` honors JSON Schema semantics: an absent `additionalProperties` is lenient (true); only an explicit `false` produces a strict object. Authored-default → `false` → round-trips. |
| **(In-flight #4) Error message strategy** | Reuse the existing `Converters.object` / `Converters.strictObject` field-name-aware reporting; no parallel error system | The object converter already prefixes each field failure with the property name. The adapter adds per-property context (`${key}: ${msg}`) for schema-translation failures and otherwise lets the underlying converters report at convert-time. No new error machinery. |
| **(In-flight #5) `fromJson` out-of-subset cutoff** | **Fail** on compositional/assertive keywords (`$ref`, `oneOf`, `anyOf`, `allOf`, `not`, `if`, `then`, `else`, `pattern`), union `type` arrays, non-string / empty `enum`, tuple-form `items`, schema-valued `additionalProperties`, and missing/unknown `type`. **Pass through (ignore)** pure annotations with no validation semantics — draft-07 `format`, `title`, `default`, `examples`, and a spurious `description` on any node. | Structural/assertive keywords cannot be honored faithfully; silently dropping them would produce a looser converter than the schema describes — the exact "illusion of safety" the alignment stream exists to avoid. Draft-07 `format` is officially an annotation (validators MAY ignore), so ignoring it is spec-compliant pass-through, not false-safety. `pattern` is always assertive → fail. Failures carry a JSON-pointer-style path (`#/properties/foo: ...`). |
| Phantom carrier: optional `__staticType?: T` field, not a module-private `unique symbol` | Implementation detail (not one of the 5 calls) | This is a **published** package surface. A module-private `unique symbol` keying the phantom property cannot be named in the emitted `.d.ts` (TypeBox issue #679, flagged as the fallback in the feasibility appendix), which would break declaration emission / API Extractor for downstream consumers. An optional, never-assigned property carries `T` for `Static` extraction with no runtime field and clean `.d.ts` emission. The `_type` discriminant remains the structural gate. |
| **(Revision) Schema IS the validator** | `ISchemaValidator<T>` extends `Validator<T>`; `toConverter.ts` and `toJson.ts` deleted | The v1 design had three entities (schema, converter, JSON emission) requiring callers to import and chain. The revised design collapses schema and validator: each factory returns an `ISchemaValidator<T>` instance with `validate(input)` and `toJson()` methods. This matches the "schema IS the validator" idiom used in other typed-schema libraries and is simpler for consumers. `fromJson` returns `ISchemaValidator<JsonValue>` (opaque; honest supertype covering all JSON value kinds). |
| **(Revision) `fromJson` enum dispatch** | Initially: `Converters.oneOf([_enumArm, _typeDiscriminatedArm])` — enum arm leads; **superseded by L1 fix**: explicit `if ('enum' in raw)` pre-flight in `jsonSchemaConverter` (Copilot round-1 L1 fix moved enum out of `oneOf` because failures fell through misleadingly to the type-dispatched arm giving "no matching converter" messages). | Non-enum nodes have a `type` field so `discriminatedObject` covers them; enum nodes have no `type` (only `enum`). The pre-flight in `jsonSchemaConverter` checks `'enum' in raw` before the `_typeDispatchConverter` dispatch, routing enum nodes directly to `_enumArm` so validation failures propagate immediately rather than falling through. |
| **(P2 fix) `fromJson` path-threading — option (a) chosen** | `Converter<ISchemaValidator<JsonValue>, string>` context carries the JSON Pointer path through all arms | Option (a) (context-typed Converter) was chosen over option (b) (inner parameter via wrapper functions). `discriminatedObject` and `oneOf` already thread `TC` context to arms (verified in `basicConverters.ts`); no ts-utils combinator changes required. Each arm function gains `(_self, context?)` parameters; `jsonSchemaConverter` defaults `context ?? '#'` and prefixes ALL its own pre-flight errors with the path; `_parseObjectBody` passes `${path}/properties/${key}` as context instead of using `withErrorFormat` (which caused double-prefixing at depth > 1). `_convertArray` similarly threads `${path}/items`. Net: every error in the parse graph names the exact failing node. |
| **(Revision) `ObjectSchemaValidator` uses `Converters.object` internally** | `_buildObjectConverter` returns `Converter<ObjectStatic<P>>`, stored as `_converter`, `validate()` overridden | `Validators.object` is in-place (passes extra properties through on success). Object schemas in lenient mode (`additionalProperties: true`) must still only surface declared properties in the validated value — callers rely on the TypeScript type `ObjectStatic<P>` to be accurate at runtime. Using `Converters.object` ensures a new output object with only declared fields is produced. The same override pattern is used for non-strict `NumberSchemaValidator`. |
| **(L3 fix) `fromJson` return type** | `ISchemaValidator<JsonValue>` | `JsonValue = JsonPrimitive \| JsonObject \| JsonArray` is the honest supertype for a runtime-parsed schema (the validated value may be a string, number, boolean, array, OR object — not just `JsonObject`). This eliminates the round-trip `as unknown as ISchemaValidator<JsonObject>` casts in all arm bodies. Internal arms still use `as unknown as ISchemaValidator<JsonValue>` narrowing casts because TypeScript's validator interface is invariant, but this is now phantom-type-only and the public contract is accurate. No external consumers of `fromJson` exist in this repo. |
| **(L2 fix) Arm rewrite strategy** | `Converters.field` / `Converters.optionalField` + direct dispatch for enum arm | Per-arm `Converters.object` was considered but creates TC-type mismatch complications (TC=string for the path context vs TC=unknown for field extractors). Using `Converters.field` / `Converters.optionalField` for individual field extraction eliminates the `from as Record<string, unknown>` anti-pattern while keeping TC threading clean. The enum arm was moved from `oneOf` dispatch to direct dispatch in `jsonSchemaConverter` pre-flight — `oneOf` caused validation failures to fall through to the type-dispatched arm (giving misleading "no matching converter" messages). |
| **(L4 fix) `OptionalSchemaValidator` unsafe cast** | `result as Failure<Static<S> | undefined>` (single phantom cast) | `inner.validate(from)` returns `Result<unknown>` (TypeScript infers `unknown` from `S extends ISchemaValidator<unknown>`). On failure, `Failure<unknown>` is structurally identical to `Failure<Static<S> \| undefined>` (Failure carries only a message string; T is phantom). The original `result as unknown as Failure<T>` double-cast is replaced by a direct single phantom cast. A fully cast-free alternative (overriding `validate()`) was considered but requires the same phantom cast in `onSuccess`. |

| **(R3 fix) validate/convert symmetry — shape (a) chosen** | Each schema class overrides both `validate()` and `convert()` routing through the same internal Converter or Validator | Shape (a): override both methods on each class. Shape (b): override only `convert()` and have `validate()` delegate to it. Shape (a) was chosen because `ISchemaValidator<T>` extends `Validator<T>`, and `validate()` is conceptually the canonical operation (in-place validation); making `validate()` a thin wrapper over `convert()` would be semantically inverted for classes where the input does equal the output (e.g. `StringSchemaValidator`). String/boolean/enum schemas have `validate() === convert()` (same result object); number and object schemas have a genuine conversion (coercion / property stripping) and both methods route to the same internal Converter. `OptionalSchemaValidator` is the only class where `validate()` delegates to `convert()` — deliberately, so that optional fields propagate inner transformations on both call sites. |
| **(R3 fix) `OptionalSchemaValidator` routes through `inner.convert()`** | `convert(from)` calls `inner.convert(from)`; `validate(from)` calls `this.convert(from)` | The previous `validate()` override called `inner.validate(from)` and returned `true` on success, discarding the converted value. This worked for leaf schemas where validate-in-place is correct, but caused non-strict number coercion and object property stripping to be lost when optional wrappers were involved. By routing through `inner.convert()`, the optional wrapper preserves all transformations from the inner schema. The undefined/absent case returns `succeed(undefined)` directly without touching the inner schema. |
| **(R3 fix) `_descriptionField` behavior documented** | Docstring tightened: non-string `description` fails (not ignored) | The previous docstring said "non-string description is silently ignored." The actual `Converters.optionalField('description', Converters.string)` implementation FAILS if the field is present with a non-string value. The design intent for R3-L5 was to tighten the docstring to match (not relax the implementation). A non-string `description` is a schema authoring error that should surface — silently ignoring it would hide bugs. |
