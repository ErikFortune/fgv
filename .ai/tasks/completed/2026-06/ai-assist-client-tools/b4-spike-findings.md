# B4 Spike Findings — Anthropic Thinking + Tools Round-Trip

**Stream:** ai-assist-client-tools  
**Phase:** B — empirical spike  
**Date:** 2026-06-04  
**Status:** Complete — verdict reached from documentation and SDK evidence; no live API call required (no key available in agent environment)  
**Question answered:** Does the Anthropic Messages API round-trip continuation require thinking-block preservation in the assistant turn when responding to a `tool_result` input block, and if so, what is the exact shape requirement?

---

## §1 — Verdict

**Yes — thinking block preservation in the reconstructed assistant turn is a hard API requirement, not advisory.**

When extended thinking is active and the model emits `tool_use` content blocks (client tool calls), the assistant message for the follow-up request must include all `thinking` and `redacted_thinking` blocks from the original assistant turn, in their original positions, completely unmodified (including their cryptographic `signature` field). Omitting these blocks, stripping the `signature` field, or reordering blocks relative to each other causes Anthropic's API to return an HTTP 400 error. The exact error text is:

> "messages.N.content.0.type: Expected `thinking` or `redacted_thinking`, but found `text`. When `thinking` is enabled, a final `assistant` message must start with a thinking block (preceding the lastmost set of `tool_use` and `tool_result` blocks). We recommend you include thinking blocks from previous turns. To avoid this requirement, disable `thinking`."

The ordering constraint is strict: within an assistant message, `thinking` (or `redacted_thinking`) blocks must appear **first**, before any `text` or `tool_use` blocks. With interleaved thinking (discussed in §5), the interleaved sequence itself must also be preserved unmodified.

The round-trip continuation requirement affects both the **streaming adapter (C2)** and the **continuation builder (C3)**: C2 must accumulate thinking blocks from the original stream; C3 uses those accumulated blocks to reconstruct the assistant message for the follow-up request.

### E4 — `redacted_thinking` block type and its round-trip requirements

Source: multiple community sources corroborating official documentation behavior, including LiteLLM PR [BerriAI/litellm#15501](https://github.com/BerriAI/litellm/pull/15501) and web search synthesis (2025 ecosystem).

From the documentation and community sources:

> "If your code filters content blocks by type (for example, `block.type == "thinking"`) when round-tripping responses with tool use, also include `redacted_thinking` blocks. Filtering on `block.type == "thinking"` alone silently drops `redacted_thinking` blocks and breaks the multi-turn protocol."

The `redacted_thinking` block appears when Anthropic's safety systems redact a portion of the model's thinking. Its wire format is:

```json
{
  "type": "redacted_thinking",
  "data": "<opaque encrypted string>"
}
```

Unlike a `thinking` block (which has `thinking` text + `signature`), a `redacted_thinking` block has only an opaque `data` field. Both must be passed back to the API unmodified during round-trips.

### E5 — Streaming format for thinking blocks (how to accumulate them)

Source: [GitHub issue openclaw/openclaw#87574](https://github.com/openclaw/openclaw/issues/87574) and web search results (2025).

The streaming sequence for a thinking block:

1. `content_block_start` event: `{ "type": "content_block_start", "index": N, "content_block": { "type": "thinking", "thinking": "" } }`
2. One or more `content_block_delta` events with `delta.type === "thinking_delta"`: the `thinking` text must be **appended** across deltas.
3. One or more `content_block_delta` events with `delta.type === "signature_delta"`: the `signature` must be **appended** across deltas — not overwritten. The signature is base64-encoded and arrives in multiple chunks.
4. `content_block_stop` event.

The accumulated block for round-trip use is `{ "type": "thinking", "thinking": "<full text>", "signature": "<full signature>" }`.

A critical implementation bug documented by the community (openclaw#87574): if `signature_delta` chunks are overwritten instead of appended, the signature is truncated, and subsequent round-trip requests fail with Anthropic's 400 error.

For `redacted_thinking` blocks in the stream: the `content_block_start` event has `"type": "redacted_thinking"` and a `"data"` field. No `thinking_delta` events follow. The round-trip block is `{ "type": "redacted_thinking", "data": "<opaque>" }`.

### E6 — Interleaved thinking (Claude 4 models)

Source: [openclaw/openclaw#52889](https://github.com/openclaw/openclaw/issues/52889), web search synthesis on interleaved thinking.

With interleaved thinking (enabled via `anthropic-beta: interleaved-thinking-2025-05-14` on older Claude 4 models; automatically active on Opus 4.6 and Sonnet 4.6 with adaptive thinking), the assistant can emit multiple thinking blocks interleaved with multiple tool_use blocks within a single turn:

```
[thinking, text, tool_use, thinking, text, tool_use]
```

The exact error when interleaved blocks are reordered (GitHub issue openclaw#52889):

> "thinking or redacted_thinking blocks in the latest assistant message cannot be modified. These blocks must remain as they were in the original response."

The fix — preserving original interleaved order (from the issue fix):

```javascript
// CORRECT: preserve original interleaved order
const nextContent = [];
for (const block of content) {
  if (isThinkingOrRedactedBlock(block)) {
    nextContent.push(block); // preserve as-is, in position
  } else {
    const [sanitized] = await sanitizeContentBlocksImages([block], label, imageSanitization);
    nextContent.push(sanitized);
  }
}

// WRONG: consolidates all thinking blocks first — API rejects this
const thinkingBlocks = content.filter(isThinkingOrRedactedBlock);
const otherBlocks = content.filter(b => !isThinkingOrRedactedBlock(b));
const nextContent = [...thinkingBlocks, ...otherBlocks]; // API rejects
```

## §3 — Impact on C3 Continuation Builder

### 3.1 How the adapter must accumulate thinking blocks

The Anthropic streaming adapter (`anthropic.ts`, extended in C2) must track an **ordered accumulation buffer** of all content blocks from the original assistant turn, not just text and tool_use blocks. The buffer must preserve:

- `thinking` blocks: accumulated from `thinking_delta` events (text appended), with `signature` accumulated from `signature_delta` events (also appended, not overwritten). Complete wire block: `{ type: 'thinking', thinking: '<full>', signature: '<full>' }`.
- `redacted_thinking` blocks: passed through as-is from `content_block_start`; the `data` field is opaque and must not be modified.
- `text` blocks: accumulated from `text_delta` events (already handled by existing adapter).
- `tool_use` blocks: accumulated from `input_json_delta` events (C2 adds this for client tools).

**The ordering of blocks in the accumulation buffer must match the ordering in the stream.** The buffer is positionally ordered by `index` from `content_block_start` events. The C3 builder reconstructs the assistant message using this ordered buffer exactly.

### 3.2 Wire format of the follow-up assistant turn

The follow-up request's messages array must include the original assistant turn reconstructed from the ordered accumulation buffer:

```json
{
  "role": "assistant",
  "content": [
    {
      "type": "thinking",
      "thinking": "<accumulated thinking text>",
      "signature": "<accumulated signature — complete and unmodified>"
    },
    {
      "type": "text",
      "text": "<any text the model emitted before the tool call>"
    },
    {
      "type": "tool_use",
      "id": "toolu_01abc",
      "name": "recall_memory",
      "input": { "query": "last preference" }
    }
  ]
}
```

Followed by the new user turn with tool results:

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01abc",
      "content": "User prefers dark mode"
    }
  ]
}
```

With parallel tool calls (multiple `tool_use` blocks in the same turn), all `tool_use` blocks go into the reconstructed assistant turn in their original stream positions, and all results go into the new user turn as an array of `tool_result` blocks.

### 3.3 Ordering constraints on thinking vs tool_use blocks

**Non-interleaved (standard extended thinking):** the ordering within the original assistant turn is `thinking → [text] → tool_use`. The `thinking` block is always position 0 in the content array. The C3 builder must not reorder this.

**Interleaved thinking (Claude 4 models / adaptive thinking):** the ordering is `thinking → [text →] tool_use → thinking → [text →] tool_use ...`. Each `thinking` block belongs to the `tool_use` block it precedes in stream order. The C3 builder must preserve the full interleaved sequence exactly as the stream emitted it.

### 3.4 Whether this affects the streaming adapter (C2) or only the continuation builder (C3)

This affects **both** C2 and C3:

**C2 (streaming adapter):** Must be extended to accumulate thinking and redacted_thinking blocks in the ordered buffer, alongside tool_use blocks. Specifically:
- Handle `content_block_start` with `type === 'thinking'` and `type === 'redacted_thinking'`.
- Handle `content_block_delta` with `delta.type === 'thinking_delta'` (append to thinking text buffer).
- Handle `content_block_delta` with `delta.type === 'signature_delta'` (append — not overwrite — to signature buffer).
- The current adapter ignores these events (comment: "other values (e.g. `input_json_delta` for tool args) are ignored"). C2 must explicitly handle them.

**C3 (continuation builder):** Must read the ordered accumulation buffer from C2 — including thinking and redacted_thinking blocks — and emit all blocks in their original stream order when constructing the reconstructed assistant turn. A naive implementation that only accumulates text + tool_use blocks produces an API 400 error when thinking is active.

## §5 — Edge Cases / Open Items

### 5.1 Multiple thinking blocks interleaved with multiple tool_use blocks

With interleaved thinking (active on Opus 4.6 and Sonnet 4.6 with adaptive thinking), one assistant turn can contain:
```
[thinking_1, text_1, tool_use_1, thinking_2, text_2, tool_use_2]
```

The positional accumulation buffer handles this correctly — emit all blocks in index order. The C3 builder must not attempt to separate, consolidate, or reorder thinking blocks.

The API error when reordering occurs is: "thinking or redacted_thinking blocks in the latest assistant message cannot be modified. These blocks must remain as they were in the original response." (source: openclaw#52889)

### 5.2 Parallel tool calls with interleaved thinking

**Docs are ambiguous here.** If the model issues two tools in parallel (two `tool_use` blocks at the same logical position), it is unclear whether interleaved thinking emits one thinking block before both `tool_use` blocks or one per `tool_use` block. The stream's positional `index` values will differentiate them; the accumulation buffer handles both cases by preserving whatever order the stream produces.

**Recommendation:** flag this for live verification in the Phase C testbed scenario. The positional accumulation buffer design is safe either way; no special-casing is needed.

### 5.3 Signature accumulation correctness is critical

The `signature` field on a `thinking` block is a cryptographic attestation. Three implementation hazards:

1. **Signature_delta chunks must be appended, not overwritten.** The signature is base64 and arrives across multiple `signature_delta` events. Overwriting on each event (the most obvious mistake) leaves only the last chunk; the API rejects the truncated signature. (Documented production bug: openclaw#87574.)
2. **Signature must not be re-encoded.** Any JSON re-encode/re-decode path that normalizes base64 whitespace or line endings will corrupt it.
3. **Signature must not be omitted.** A `thinking` block with `thinking` text but no `signature` field is rejected.

### 5.4 `tool_choice` constraint

When `resolvedThinking` is set (thinking active), the follow-up request from C3 must not include `tool_choice: { type: 'any' }` or `tool_choice: { type: 'tool', name: '...' }`. The existing `callAnthropicStream` never sets a forced `tool_choice` — the continuation builder must follow the same convention for follow-up requests.

### 5.5 `redacted_thinking` blocks must be included alongside `thinking` blocks

Filtering on `block.type === 'thinking'` alone silently drops `redacted_thinking` blocks. The C2 accumulation buffer must track both types explicitly. The `redacted_thinking` wire format (`{ type: 'redacted_thinking', data: '...' }`) differs from `thinking` (`{ type: 'thinking', thinking: '...', signature: '...' }`); the C3 builder must emit the correct shape for each.

### 5.6 Anthropic docs inaccessible via direct URL during this spike

`docs.anthropic.com` returned HTTP 403 consistently. The `platform.claude.com/docs` URL was partially accessible (one fetch succeeded for the extended thinking page). The evidence in §2 combines: (a) the successfully fetched official docs page with quoted Python example and wire format, (b) verbatim API error responses captured in GitHub issues (the errors originate from Anthropic's server, not user text), and (c) implementation bugs from major open-source AI frameworks that reproduce the behavior. The verdict is well-evidenced despite the docs fetch limitation.

## §6 — Phase C Implementation Recommendation

The C3 brief should instruct the implementing agent as follows:

The Anthropic-side continuation builder in Phase C must treat the C2 accumulation buffer as a **complete ordered record of all content blocks from the original assistant turn**, not just `text` and `tool_use` blocks. The buffer must accumulate `thinking` blocks — with `thinking` text concatenated across `thinking_delta` events, and `signature` concatenated across `signature_delta` events (appended, never overwritten; the signature is base64 arriving in multiple chunks) — and `redacted_thinking` blocks (passed through opaquely from `content_block_start`, no delta events). When building the follow-up request's assistant message, the buffer must be emitted in original stream order (positional, by SSE `index`), with thinking and redacted_thinking blocks in their exact positions relative to text and tool_use blocks. Stripping, reordering, or consolidating thinking blocks causes the Anthropic API to return HTTP 400. The wire format for `thinking` is `{ type: 'thinking', thinking: '<full text>', signature: '<full base64 — never truncated>' }`; for `redacted_thinking` it is `{ type: 'redacted_thinking', data: '<opaque>' }`. Both are passthrough-only — never modified, never re-encoded. The Phase C testbed scenario must exercise thinking + client tool use on Anthropic end-to-end before the C3 PR is opened; signature verification is server-side and cannot be confirmed by unit tests alone. Additionally, the continuation builder must not set `tool_choice: { type: 'any' }` or `tool_choice: { type: 'tool', ... }` on follow-up requests when thinking is active.
