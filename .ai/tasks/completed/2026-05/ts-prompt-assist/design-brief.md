## What we're building

A new TypeScript library in the `@fgv/*` family that provides **ts-res-driven prompt resolution with Mustache substitution** as a reusable capability. Consumers bring their own scope model, their own actor types, their own prompt content; the library provides the resolution machinery, the type system, the descriptor schema, the storage abstraction, and the LLM-prompt-specific safeguards.

**Package name**: `@fgv/ts-prompt-assist` (standalone — see "Why a separate package" below).

**Mental model**: every consumer surface that calls an LLM with a prompt does two things — **lookup** (pick the right variant via qualifier-conditioned resolution) and **compose** (fill the chosen body's blanks with this caller's content). The library standardizes both halves. Surface-specific composition downstream of the body (chat-message wrapping, image-API call, classifier post-processing) stays with the consumer.

## Why now

The pattern emerged from one consumer's design discussions and the observation that two divergent in-codebase prompt libraries had already reinvented the same machinery with subtly different policies. Concrete pain shapes driving the abstraction:

- **Hard to enforce a consistent visual / brand style** across surfaces — each surface re-implements its own framing approach; tenant-level style overrides have no canonical home.
- **Variation prompts are hit-or-miss** — qualifier-conditioned variants (tone, time-of-day, region) are hand-rolled per surface, with inconsistent fallback behavior when no variant matches.
- **Hardcoded behavior in call sites** that won't scale as more surfaces and more tenant-level customization come online.

Generalizing the resolution machinery surfaces a reusable primitive, gives operators a single mental model for "where do I edit this," and prevents the same drift across future consumers.

## Why a separate package

`@fgv/ai-assist` (the AI conversation orchestration library) lives in `@fgv/ts-extras`. `ts-prompt-assist` depends on `@fgv/ts-res`, which sits **above** `ts-extras` in the dependency graph. Folding prompt-assist into `ts-extras` would create a cycle. The library therefore ships as its own package above `ts-res`.

## Design principles

1. **The library is shape-agnostic about the consumer's domain.** ScopeKey is opaque (the consumer encodes its own hierarchy into strings). Surfaces, slot kinds, and slot sources are open-string fields the consumer narrows via descriptor Converter at load time. Closed-vocabulary registrations (e.g., a closed enum used as a qualifier value set) live in the shape registry the consumer populates at boot.
2. **Lookup is uniform; composition is plural.** ts-res handles qualifier-conditioned candidate selection. Mustache handles substitution. After substitution, the resolved body is the consumer's to wrap into chat messages / image-API calls / classifier dispatches / whatever — that's not the library's concern.
3. **Resource bindings make hierarchical composition a first-class case.** A slot binding can be a literal value OR a reference to another resource. The library resolves the inner resource recursively (with cycle detection and a depth cap). Without this, consumers have to hand-orchestrate sub-prompt resolution and the resource dependency graph is invisible.
4. **Qualifier metadata, not qualifier schema.** Descriptors declare `required` / `expected` / `disallowed` axes but never a closed enum of values per axis. New axes register at scope level via dedicated files. ts-res's open-ended fallback semantics are the substrate; closing them defeats the point.
5. **Output contracts are typed and validated by the library.** Descriptors declare `output: { kind: 'json', converterId }`; the library strips fences, parses JSON, runs a registered Converter, and runs registered cross-shape validators. Consumers stop hand-rolling this per surface.
6. **Storage-agnostic via `IPromptStore`.** FileTree adapter (per the existing `filetree-io` convention) is canonical for v0.1; in-memory for tests; SQL/Mongo adapters drop in later without library changes. The store interface deliberately doesn't know about file paths or YAML — those are FileTree adapter details.

## Scope: what's in, what's out

### In scope (library v0.1)

| Capability | Notes |
|---|---|
| `IPromptDescriptor` type system | Brand types, string-union types, descriptor / slot / binding / output-contract / qualifier-metadata shapes — see §"Type system" |
| `PromptLibrary` resolution class | `resolve(req)`, `resolveAndValidateOutput<T>(req, raw)`, scope-chain walker (generic over scope-key shape) |
| `IPromptStore` interface + adapters | `FileTreePromptStore` (canonical, per `filetree-io`), `InMemoryPromptStore` (tests) |
| YAML descriptor format + Converter | Self-describing records that round-trip cleanly through FileTree / SQL / Mongo / API |
| Mustache integration | Triple-brace canonical (`{{{name}}}`, unescaped); `validateAndRender` via `@fgv/ts-extras`'s `MustacheTemplate`; cached parsed templates |
| Slot bindings with directives | `SlotBinding = literal \| resource` discriminated union; `directive: constraint \| hint \| prose`; per-binding `enforced` lock; per-slot `writableBy` tier |
| Scope-level binding records | `IScopeSlotBindingsRecord` keyed by `(scope-key)`; library merges across the chain at resolve time, most-specific wins (except enforced) |
| Open qualifier metadata | `required` / `expected` / `disallowed` per descriptor; axis registration via scope-level files |
| Output post-processing | Strip code fences, parse JSON, run Converter, run registered validators by id |
| Resource-binding resolution | Recursive resolve with cycle detection; configurable depth cap (default 5) |
| `PromptShapeRegistry` | Registered Converters by id + closed-vocabulary qualifier value enums by name |
| Input-safeguard primitives | Length caps, regex screening hook (mechanism in library; policy supplied by consumer) |
| Branded scalar types + Converters | `PromptId`, `SlotName`, `ResourceId`, `ConverterId`, `SerializerId`, `AxisName`, `ScopeKey` |
| Test utilities | InMemory store, fixture builders, descriptor↔body coherence validators |

### Out of scope (v0.1)

| Concern | Where |
|---|---|
| SQL / Mongo store adapters | Future — interface designed to make these drop-in |
| Editor UI helpers | Consumer's app |
| Hot-reload via `watch` | Optional method on the store interface; v0.1 may stub |
| Schema-version migration | v1 only for now; v2 migration story when v2 lands |
| Consumer-specific surfaces / slot kinds / sources / scope hierarchies | All consumer-side; library exposes open types + registration hooks |
| Anti-jailbreak text content | Consumer supplies per-surface framing; library exposes the injection seam |
| LLM-call orchestration (provider, model, retry, cost) | Consumer's runtime |

## Type system

Canonical types — branded identifiers, string-union types, interfaces, discriminated unions. Per repo conventions, every string-union ships `allFooValues` + a Converter.

```ts
import { Brand, Converter, Logging, Result } from '@fgv/ts-utils';

// ── Branded scalars ───────────────────────────────────────────────────
export type PromptId      = Brand<string, 'PromptId'>;
export type SlotName      = Brand<string, 'SlotName'>;
export type ResourceId    = Brand<string, 'ResourceId'>;
export type ConverterId   = Brand<string, 'ConverterId'>;
export type SerializerId  = Brand<string, 'SerializerId'>;
export type AxisName      = Brand<string, 'AxisName'>;

/** Opaque scope identifier supplied by the consumer. Library doesn't
 *  interpret the format; just compares for equality and walks an
 *  ordered chain. Consumers encode their own hierarchy into strings
 *  (e.g. 'global', 'tenant:acme', 'role:editor:tenant:acme',
 *  'actor:user-42'). */
export type ScopeKey      = Brand<string, 'ScopeKey'>;

// ── String-union types (closed within the library) ────────────────────
export type PromptCompositionMode = 'single-best' | 'concat-fragments';
export type SlotBindingKind       = 'literal' | 'resource';
export type SlotDirective         = 'constraint' | 'hint' | 'prose';
export type SlotWritability       = 'any-scope' | 'schema-only' | 'system-only';

// Open within the library — consumers narrow via descriptor Converter:
//   surface       (e.g. 'chat' | 'classifier' | 'image' | 'free-text' for an LLM consumer)
//   slot.kind     (e.g. 'string' | 'json' | 'list' | 'card' for an LLM consumer)
//   slot.source   (e.g. 'editorial' | 'user-input' | 'system' for an LLM consumer)

// ── Output contract ───────────────────────────────────────────────────
export type FreeTextOutputKind  = 'free-text';
export type JsonOutputKinds     = 'json';        // poised to grow: 'json-array', 'json-stream'
export type OutputContractKinds = FreeTextOutputKind | JsonOutputKinds;

export interface ITextOutputContract {

}

  readonly kind: TKIND;
  readonly converterId: ConverterId;
}

// ── Slot bindings ─────────────────────────────────────────────────────
export interface ILiteralSlotBinding {
  readonly kind: 'literal';
  readonly value: string;
  readonly directive: SlotDirective;
  /** Descendants cannot override this binding. Default false. */

}

export interface IResourceSlotBinding {
  readonly kind: 'resource';
  readonly resourceId: ResourceId;
  /** Optional qualifier overrides for the inner resolve. Default:
   *  inherit parent. */
  readonly qualifiers?: IQualifierContext;
  /** Optional scope-chain override for the inner resolve. Default:
   *  inherit parent. */
  readonly scopeOverride?: ReadonlyArray<ScopeKey>;
  /** Optional substitution map for the inner resolve. When supplied,
   *  these subs flow into the referenced resource's resolution instead
   *  of (or in addition to — see §"Resource-binding semantics") the
   *  outer call's subs. Enables the "share a prompt fragment, call
   *  it with different values per use" pattern without forcing the
   *  inner resource to be aware of its caller. */
  readonly substitutions?: PromptSubstitutions;
  readonly directive: SlotDirective;
  readonly enforced?: boolean;
}

export type SlotBinding = ILiteralSlotBinding | IResourceSlotBinding;

/** Caller-supplied substitutions. A bare string is sugar for
 *  { kind: 'literal', value, directive: 'prose' }. */
export type PromptSubstitutions = Readonly<Record<string, string | SlotBinding>>;

// ── Qualifier metadata (open variation; ts-res fallback preserved) ────
export interface IPromptQualifierMetadata {
  /** Axes whose match is required for this resource to resolve. Library
   *  validates at resolve time. In practice rare — ts-res's fallback
   *  semantics mean most resources have a default candidate matching
   *  empty conditions, so resolution always succeeds. */
  readonly required?: ReadonlyArray<AxisName>;

  /** Axes the editor surface should suggest when authoring variants of
   *  this resource. Hints, not enforcement — authors can use any axis
   *  registered in the resolve chain, even if not listed here. */
  readonly expected?: ReadonlyArray<IExpectedQualifierAxis>;

  /** Axes the library strips from the resolve context before delegating
   *  to ts-res. Stripping is logged. Use cases: prevent sensitive axes
   *  from influencing resolution (e.g. user region), block axes
   *  irrelevant to a specific surface. */
  readonly disallowed?: ReadonlyArray<AxisName>;
}

export interface IExpectedQualifierAxis {
  readonly name: AxisName;
  readonly description?: string;
  /** Non-exhaustive hints. Editor surfaces these as quick picks but
   *  allows free entry. */
  readonly suggestedValues?: ReadonlyArray<string>;
}

export type IQualifierContext = Readonly<Record<string, string>>;

// ── Slot ──────────────────────────────────────────────────────────────
export interface IPromptSlot {
  readonly name: SlotName;
  readonly description: string;
  /** Whether resolution requires a value. Omitted = true. When false
   *  and no value is supplied (no binding, no caller sub, no default),
   *  the library substitutes empty string. */
  readonly required?: boolean;
  /** Schema-supplied default binding. Functionally a binding shipped
   *  with the descriptor; the directive carries framing intent. */
  readonly defaultBinding?: SlotBinding;
  /** Wire shape of the slot's runtime value. Open registry — consumers
   *  register their own kinds with serializers. Default: 'string'. */
  readonly kind?: string;
  /** For non-string kinds, names a registered serializer that produces
   *  the substituted string from the typed value. */
  readonly serializerId?: SerializerId;
  /** Which directive shapes the slot's binding can carry. Library
   *  rejects bindings whose directive isn't in this set. Default: any. */
  readonly allowedDirectives?: ReadonlyArray<SlotDirective>;
  /** Who's allowed to supply a binding for this slot. Default 'any-scope'.
   *    - 'any-scope':   any scope (subject to per-binding `enforced` lock)
   *    - 'schema-only': only the descriptor's `defaultBinding`; no
   *                      scope-level binding accepted
   *    - 'system-only': set by the runtime, never by an operator. The
   *                      descriptor declares no default; the call site
   *                      supplies the value.
   *  Library rejects writes that violate writability. */
  readonly writableBy?: SlotWritability;
  /** Length cap for input safeguard. Policy default applies otherwise. */
  readonly maxLength?: number;
  /** Open string. Editor hint; also drives screening behavior — the
   *  safety policy declares which sources are screen-eligible. */
  readonly source?: string;
}

  readonly examples?: ReadonlyArray<IPromptExampleSet>;
```

```ts
  readonly scope?: ScopeKey;

// ── Library class ─────────────────────────────────────────────────────
export interface IPromptLibraryCreateParams {
  readonly store: IPromptStore;
  readonly shapeRegistry?: IPromptShapeRegistry;
  readonly slotKindRegistry?: ISlotKindRegistry;
  readonly outputValidations?: IPromptOutputValidationRegistry;
  readonly safetyPolicy?: IPromptSafetyPolicy;
  readonly logger?: Logging.ILogger;
  /** Default 5. Resource bindings that exceed this depth fail loudly. */
  readonly resourceBindingDepthLimit?: number;
  /** Default 256. LRU cap on parsed Mustache templates. */
  readonly templateCacheSize?: number;
}

export class PromptLibrary {
  public static create(params: IPromptLibraryCreateParams): Promise<Result<PromptLibrary>>;

  /** Returns the descriptor for an id. Source of truth for editor UIs,
   *  validators, anything that reasons about a prompt's shape without
   *  resolving content. */
  public describe(id: PromptId): Result<IPromptDescriptor>;

  /** Lookup-then-compose.
   *
   *  Walks `chain` most-specific-first; the first scope with a record
   *  for `id` wins. Within the winning record, qualifier-conditioned
   *  candidates resolve per descriptor.compositionMode:
   *    - 'single-best' — ts-res's best-match candidate
   *    - 'concat-fragments' — all matching candidates, joined
   *      least→most-specific with newline separators
   *
   *  Bindings merge across the chain (most-specific wins, except where
   *  a higher-scope binding has `enforced: true`). Caller-supplied
   *  substitutions override merged bindings (except enforced).
   *  Resource bindings resolve recursively with cycle detection.
   *
   *  After substitution context is built, the body is rendered via
   *  Mustache (triple-brace tokens, unescaped). Returns the composed
   *  body and a full trace. */
  public resolve(req: IPromptResolveRequest): Promise<Result<IResolvedPrompt>>;

  /** Resolve + post-process raw LLM output through the descriptor's
   *  output contract. For kind:'json':
   *    1. Strip code fences
   *    2. JSON.parse
   *    3. Run registered Converter by output.converterId
   *    4. Run registered outputValidations[] by id
   *  For kind:'free-text': returns the raw output verbatim. */
  public resolveAndValidateOutput<T>(
    req: IPromptResolveRequest,
    rawOutput: string
  ): Promise<Result<T>>;
}

  readonly id: PromptId;
  /** Most-specific to most-general. Library walks in order; consumer
   *  builds the chain from its own domain types. */

  readonly qualifiers: IQualifierContext;
  readonly substitutions?: PromptSubstitutions;
}

  readonly id: PromptId;

  readonly body: string;
  readonly descriptor: IPromptDescriptor;
  /** Rich enough for an editor to render "where did this value come
   *  from" / "which scope's record won" / "what bindings merged". */
  readonly trace: IPromptResolveTrace;
}

export interface IPromptResolveTrace {
  readonly winningScope: ScopeKey;
```

## YAML descriptor format

One file per `(scope, prompt-id)`. Self-describing — the descriptor and candidates live together so a record round-trips through any backend without external metadata.

```yaml
# Example: <root>/<scope-encoding>/assistant.greeting.yaml
schemaVersion: '1'
id: assistant.greeting
title: Assistant greeting
description: Opening line the assistant uses to greet a new conversation.
surface: chat
compositionMode: single-best
qualifiers:
  expected:
    - name: tone
      description: Conversation tone.
      suggestedValues: [formal, casual, playful]
slots:
  - name: userName
    description: Caller-supplied name; first-name basis.
    required: false
    kind: string
    allowedDirectives: [prose]
    source: user-input
    maxLength: 100
    defaultBinding:
      kind: literal
      value: there
      directive: prose
  - name: tenantContext
    description: Editorial summary of the tenant.
    required: false
    kind: string
    allowedDirectives: [prose, hint]
    source: editorial
    maxLength: 400
output:
  kind: free-text
candidates:
  - conditions: {}
    body: >-
      Hello {{{userName}}}. {{{tenantContext}}}
      How can I help today?
  - conditions: { tone: playful }
    body: >-
      Hey {{{userName}}}! {{{tenantContext}}}
      What's on your mind?
```

Scope-level binding record:

```yaml
# Example: <root>/<scope-encoding>/_bindings.yaml
scope: tenant:acme
bindings:
  tenantContext:
    kind: literal
    value: 'Acme Corp — internal tools team'
    directive: prose
  tenantBrandTone:
    kind: literal
    value: 'professional and direct'
    directive: constraint
    enforced: true
```

Scope-level qualifier axis registration:

```yaml
# Example: <root>/<scope-encoding>/_qualifiers.yaml
scope: tenant:acme
axes:
  time-of-day:
    priority: 750
    type: string
    description: Coarse time-of-day bucket for time-sensitive variants.
  region:
    priority: 800
    type: enum
    values: [na, emea, apac]
```

Note: `scope` is an opaque `ScopeKey` string. The format above (`'tenant:acme'`, `'global'`, `'actor:user-42'`) is illustrative — consumers encode their own hierarchy. The library doesn't parse it.

## Mustache rules

- **Triple-brace canonical**. Every `{{...}}` token in a body is `{{{name}}}` (unescaped). Mustache's default double-brace `{{name}}` HTML-escapes (`& → &amp;`, `' → &#39;`), which is wrong for LLM prompt content where free-text values must travel verbatim.
- **Loader rejects double-brace**. The descriptor Converter scans the body for any double-brace `{{name}}` or ampersand `{{&name}}` token and fails loudly: `prompt '<id>': body uses double-brace token '{{name}}'; use triple-brace '{{{name}}}'`.
- **`validateAndRender`**. The library uses `@fgv/ts-extras`'s `MustacheTemplate.create(body).validateAndRender(context)`. Missing-variable errors surface with the prompt id.
- **Template caching**. Parsed `MustacheTemplate` instances cached keyed by `(id, body-hash)`. LRU bounded by `templateCacheSize` (default 256).

## Resource-binding semantics

A binding can be `{ kind: 'resource', resourceId, qualifiers?, scopeOverride?, substitutions? }`. At resolve time:

1. **Recursive resolve** with the referenced resource id, inheriting the parent's scope chain (unless `scopeOverride`) and qualifier context (unless `qualifiers` override).
2. **Inner substitution context** is built fresh per resolve. If the binding supplies `substitutions`, those take precedence; otherwise the parent's substitution context flows in. The exact merge semantics are a proposal — the strict version is "binding subs replace parent subs entirely" (cleanest); a relaxed alternative is "binding subs layer over parent subs, with binding subs winning per-key." Implementor picks based on what the first port reveals; document the choice. The intent is to enable **shared prompt fragments**: one fragment used in multiple places, each caller can supply its own values without the fragment being aware of who's calling.
3. **Cycle detection**: track currently-resolving `(scope-chain-hash, id)` pairs. Cycle → fail with `prompt '<id>': cycle detected — <a> → <b> → <a>`.
4. **Depth limit**: configurable (default 5). Exceeded → fail with `prompt '<id>': resource binding depth limit exceeded`.
5. **Sub-prompt body becomes slot value**: the inner resolve's body string is what fills the parent's slot, framed per the binding's `directive`.

The `substitutions?` field on `IResourceSlotBinding` is in scope for v0.1 — anticipating that prompt-fragment sharing is a near-term need rather than a later one.

---

## Output validation pipeline

For `output.kind: 'json'`, `resolveAndValidateOutput<T>`:

1. **Strip code fences** — LLMs wrap JSON in triple-backticks despite instructions; shared post-processor handles this.
2. **JSON.parse** — failure → `Result<fail>` with prompt id + first 200 chars of raw output.
3. **Run registered Converter** by `output.converterId`. Failure → `Result<fail>` with prompt id + Converter error chain.
4. **Run registered `outputValidations[]`** — each referenced by string id; library resolves against the output-validation registry; runs in order; failures aggregate via `MessageAggregator` from `@fgv/ts-utils`.
5. **Return typed `Result<T>`**.

For `output.kind: 'free-text'`: returns raw output verbatim.

## Input safeguards

Per-call, after substitution context is built, before Mustache renders:

1. **Per-slot length cap**. Each slot may declare `maxLength`; policy default applies otherwise. Reject on overflow with structured error: `prompt '<id>': slot '<slot>' exceeds maxLength <n> (got <m>)`.
2. **Regex screening** for slot values whose `source` is in the policy's `screenedSources`. Each pattern is tried; matches collected. `onSuspicious: 'warn'` logs + proceeds; `'reject'` fails the resolve.
3. **Source-aware skipping**. Slots with `source: 'system'` (or whatever the consumer maps to "system-supplied") skip screening — those values come from the runtime, not from operator input.
4. **Anti-jailbreak framing seam**. The library exposes a per-surface text-injection hook before the resolved body. Consumer supplies the framing text per surface. Library does not ship default text.

## Composition semantics

`compositionMode` per descriptor:

- **`single-best`** — ts-res's `resolveResource` returns one candidate. Used when each candidate represents a complete body and qualifier resolution picks the winner.
- **`concat-fragments`** — ts-res's `resolveAllResourceCandidates` returns all matching; library joins least→most-specific with newline. Used when candidates are body fragments that compose into one prompt.

Substitution always runs **once on the final composed body**, never fragment-by-fragment. (Otherwise a `{{{slot}}}` token that appears in only one fragment would render at most once even if the slot value should apply across fragments.)

Scope-chain walking and intra-record qualifier resolution are independent: the chain walker picks the winning scope; within the winner, ts-res picks the winning candidate(s).

**No cross-scope merging of bodies**. If an inner-scope record has only `mood: sad` candidates and the resolver context says `mood: happy`, ts-res's fallback within the record applies (matches the empty-condition base if one exists; fails otherwise). The library does NOT fall through to a different scope to find a matching variant — operators authoring an override expect to see the full body they're composing against, not stitched-from-multiple-scopes content. (Bindings DO merge across scopes; bodies don't.)

## Repo conventions to follow

Follows established `@fgv/*` family conventions (no `any`; Result pattern end-to-end; Converters / Validators for type-safe transformation; FileTree via `@fgv/ts-json-base` for all I/O; logging via `@fgv/ts-utils`'s `logging` packlet; structural equality via `Hash.Crc32Normalizer.computeHash`; no re-exports outside barrels; TSDoc on every exported symbol). See the family CLAUDE.md.

Design-specific patterns this library introduces or relies on heavily:

- **Mustache via `@fgv/ts-extras`'s `MustacheTemplate`** with **triple-brace canonical** (`{{{name}}}`). Double-brace tokens in descriptor bodies are rejected at load time. Detail in §"Mustache rules".
- **Open string fields narrowed by descriptor Converter**: `surface`, `slot.kind`, `slot.source` are typed `string` at the library boundary; consumers supply a descriptor Converter at load time that narrows to their closed vocabulary. Detail in §"Type system".
- **Open registries for closed-vocabulary values**: qualifier-axis enums, output Converters, slot-kind serializers, output validators all register against shape-registry interfaces at boot. The library exposes the registration seam; consumers populate it.

The library is a candidate for v0.1 publish when the following are true. Treat as a proposed bar; the implementing stream and the orchestrator coordinate the final cut.

1. **Type system complete** — every type from §"Type system" implemented with `allFooValues` + Converter for each string-union, branded types validated, discriminated unions with type-guarded converters.

## Stability contract

The library promises to consumers:

- **Stable type surface within a major version**: `IPromptDescriptor`, `IPromptSlot`, `SlotBinding`, `IPromptStore` shapes don't churn within v0.x.
- **Resolve is deterministic** for a given `(store snapshot, chain, qualifiers, substitutions)` — no time-dependent behavior, no global state.
- **`Result<T>` end-to-end** — no thrown exceptions out of public methods.
- **Trace surfaces are stable** — editor implementations can rely on the `IPromptResolveTrace` shape.
- **Adapters are pluggable** — implementing `IPromptStore` is sufficient to slot a new backend in. No hidden coupling to the FileTree adapter.
- **ScopeKey is opaque** — the library never parses its content; consumers own the encoding.

## Versioning and publishing cadence

`@fgv/*` packages version in lockstep; cadence is the `@fgv/*` orchestrator's call. Expected shape:

- **Alpha series first**. The library publishes as part of the alpha series (`5.1.0-<n>` or the next series cut) once it's ready and other family packages it depends on are at compatible alphas. Consumers pin to a specific alpha during the design-and-port cycle.
- **Stable publish** lands as part of a `@fgv/*` family stable release, **after** the new published code has settled enough that no further breaking changes are expected — i.e. once a first-consumer port has pressure-tested the API and any API churn has been absorbed.

The implementation can start as soon as the alpha series can accommodate the new package; the stable cut is gated on API stability evidence, not calendar.

## Open questions

These are decisions the implementing team makes during the build; they're surfaced here so the rationale gets documented when chosen, not as gates on starting.

1. **Scope encoding flexibility.** `FileTreePromptStore` takes `scopeEncoding` / `scopeDecoding` functions. Default: treat ScopeKey as path-safe string. Cover the bases or grow richer abstraction (nested directories mirroring scope structure)? v0.1: flat default, function override is the escape hatch. Surface if the default falls down.

2. **Resource-binding substitution merge semantics.** The brief proposes "binding subs replace parent subs entirely" as the clean version, with the option to relax to "binding subs layer over parent subs, per-key" if the build surfaces a reason. Decide during implementation; document the choice.

3. **`watch()` semantics.** Hot-reload is anticipated as a future need but no concrete requirement yet. v0.1 may ship `watch?` as optional-and-unimplemented on the FileTree adapter (the InMemory adapter should support it — tests will exercise it). Revisit when a concrete consumer demands it.

Standard packaging concerns (Rush project shape, dep versions, MessageAggregator usage, TypeDoc / documentation site, schema-version handling, output-validation registry shape) follow established `@fgv/*` conventions — the family orchestrator coordinates these at packaging time.

## First-consumer pressure-test

The first consumer port — porting an existing in-codebase prompt resolution implementation to consume this library — is the design's pressure test. It surfaces API gaps and unrealistic abstractions before stable publish. Expect 1-2 iteration cycles before the API stabilizes.

The consumer brings:
- Their own scope hierarchy (encoded into `ScopeKey` strings)
- Their own closed `surface` enum (narrowed via descriptor Converter)
- Their own slot-kind serializers (registered via `ISlotKindRegistry`)
- Their own qualifier value enums (registered via `IPromptShapeRegistry`)
- Their own output Converters (registered via `IPromptShapeRegistry`)
- Their own anti-jailbreak text content (supplied via `IPromptSafetyPolicy` or surface composers)
- All of their default prompt content (as YAML files under the FileTree root)

The library cares about none of these specifics. It cares that the contract surface above is enough to express them. A clean port implies a credible contract.
