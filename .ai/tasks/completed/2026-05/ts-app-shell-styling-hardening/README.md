# `ts-app-shell-styling-hardening` — shipped

**Shipped:** 2026-05-30 via PR #429 (implementation) on integration branch `ts-app-shell-styling-hardening`; squashed to `release` via the cluster-close PR.

**Package surface:** `@fgv/ts-app-shell` (`messages` packlet — icon SVGs, `MessagesProvider`, `IMessageAction` types, `ToastItem` rendering) + README + `LIBRARY_CAPABILITIES.md`.

---

## What shipped

Hardened `@fgv/ts-app-shell` against the most common consumer misconfiguration: forgetting to add `'./node_modules/@fgv/ts-app-shell/lib/**/*.{js,jsx}'` to the Tailwind `content` array. Before this stream, omitting that line silently tree-shook every geometry utility used inside the `messages` packlet (`h-3.5`, `w-3.5`, `absolute`, `left-2`, `p-1`, `relative`, …) while theme tokens (`text-muted`, `bg-surface`) kept working — producing invisible icon buttons and a viewport-overflowing magnifying-glass overlay. Three layers landed together.

### Layer 1 — Defensive inline geometry

Catastrophic-failure sites in the `messages` packlet now carry inline `style` props alongside their Tailwind classes, so they size and position correctly even when the geometry utilities are absent:

- StatusBar header toolbar icons (Funnel, Copy, X, Check) — inline `{ width: 14, height: 14 }`.
- Per-row copy icon — inline `{ width: 12, height: 12 }`.
- Search input MagnifyingGlassIcon overlay — inline `{ position: 'absolute', left: 8, top: '50%', transform: 'translateY(-50%)', width: 14, height: 14 }`.
- Search input clear-button `×` — inline `{ position: 'absolute', right: 8, top: '50%', transform: 'translateY(-50%)' }`.
- **Search wrapper** — inline `{ position: 'relative' }` so the absolute children have a containing block even when the `relative` utility class is purged. Caught in review (without this, the absolute children escape the field).

Scope is intentionally narrow: only sites where Tailwind absence produces *catastrophic* layout failure (invisible / off-screen / page-overflowing). The Tailwind preset remains the canonical styling path.

### Layer 2 — Self-diagnosing probe in `MessagesProvider`

On mount, the provider appends a hidden probe div with class `h-[7.3215px]` (a sentinel arbitrary-value Tailwind utility — uniquely-named so it can **only** be generated by Tailwind scanning ts-app-shell's built JS), measures it via `requestAnimationFrame` + `getBoundingClientRect`, and on a `0` reading emits exactly one `'warning'`-level message via the canonical `addMessage(level, text, options?)` API:

> @fgv/ts-app-shell styles are not fully loaded. Some icons may not render correctly. See setup docs for the required Tailwind content-path entry.

The message carries an `action.href` pointing at the README's `## Setup` anchor; `ToastItem` renders it as an external link (`target="_blank" rel="noopener noreferrer"`).

**Sentinel-class design (review-driven correction):** the original probe class was `h-3.5`, but `h-3.5` is common enough that a consumer using it elsewhere in their own source would silently mask a missing ts-app-shell content path entry. The arbitrary-value `h-[7.3215px]` has no plausible alternate source — if it resolves to 7.3215px, Tailwind has scanned ts-app-shell. If it resolves to 0, it hasn't.

**StrictMode safety:** the `useRef` guard that makes the probe idempotent is set *inside* the rAF callback, not synchronously on mount. The original synchronous set was vulnerable to React.StrictMode cleanup-before-rAF: the effect runs once, sets the ref, gets cleaned up, then re-runs with the ref already set — and the rAF from the first run never fires, silently dropping the diagnostic. Setting the guard in the rAF body lets the cleanup happen between mount and measurement without poisoning the next mount.

Always-on (not dev-only) — a misconfigured production build is exactly when the consumer is least likely to be watching the console; the in-panel message is the only signal that lands.

### Layer 3 — README nudges

- Top-of-document `> ⚠️ **Required setup**` callout before the existing Tailwind snippet.
- Stable `## Setup` heading anchor (the Layer-2 message's `href` target).
- New `## Troubleshooting` section: "If icons appear missing or oversized in the messages panel, your Tailwind config is missing the `@fgv/ts-app-shell` content path. See Setup above."

### Discriminated-union refactor of `IMessageAction`

Surfaced in review: the original `IMessageAction` interface declared both `href?` and `onAction?` as optional fields, but the prose contract said "exactly one of." Refactored to a discriminated union (`IMessageHrefAction | IMessageCallbackAction`) so the "exactly one" invariant is enforced at the type level rather than left as a runtime contract. `ToastItem` discriminates on the union to render either an `<a>` or a `<button>`. Active-development surface; additive on the consumer side (existing callers using `onAction` continue to compile; new `href` callers opt into the link path).

## Review highlights

PR #429 absorbed three substantive review rounds before merging:

1. **StrictMode cleanup-before-rAF poisoning the probe guard.** Original implementation set the `useRef` guard synchronously on mount; React.StrictMode's mount → cleanup → re-mount pattern dropped the rAF entirely. Fixed by deferring the guard set into the rAF callback body.
2. **`IMessageAction` allowing both/neither.** Prose said "exactly one of `href` or `onAction`"; interface allowed both/neither. Refactored to a discriminated union.
3. **Layer 1 `relative` class purgeable too.** The original Layer 1 patch added inline geometry to the search overlay icons but left the wrapper relying on the `relative` Tailwind class. That class was itself purgeable — without it the `position: absolute` children escape the field. Fixed by inline `position: relative` on the wrapper.
4. **Probe class `h-3.5` not unique.** A consumer using `h-3.5` elsewhere in their own source would silently mask a missing ts-app-shell content path entry. Switched to the sentinel arbitrary-value `h-[7.3215px]`.

All four review threads exemplify the "small focused diff + careful review = better outcome" pattern. The discriminated-union refactor in particular elevated the change from "ship and hope callers read the prose" to "enforce at the type level."

## Origin

Cross-repo debug 2026-05-29: Erik on personaility browser at `@fgv/ts-app-shell@5.1.0-32` reported "no filter button visible" on the `StatusBar` messages panel. DOM inspection confirmed `class="h-3.5 w-3.5 text-muted"` SVGs present but unsized; clicking near them surfaced a viewport-overflowing magnifying-glass overlay. Diagnosis: personaility's `tailwind.config.js` was missing `'./node_modules/@fgv/ts-app-shell/lib/**/*.{js,jsx}'` in its `content` array, so Tailwind tree-shook every geometry utility used inside ts-app-shell while keeping the theme tokens (because personaility used those in its own source). README was correct but easy to miss; failure mode was silent and misleading enough to warrant defensive measures in the package itself.

The fix on personaility's side is the one-line content path entry, but the failure mode is bad enough that the next consumer to misconfigure would have hit it just as hard. This stream made ts-app-shell self-defending and self-diagnosing.
