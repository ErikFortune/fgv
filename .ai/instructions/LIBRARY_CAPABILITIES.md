# FGV Library Capabilities Quick Reference

**Purpose:** Before writing new utility code, scan this guide. If a capability is listed here, use the existing library тАФ do not reimplement it. Each entry links to the source on the `release` branch.

**Repository:** https://github.com/ErikFortune/fgv (release branch)

---

## How to use this guide

1. Find the capability area you need (parsing, validation, JSON, file trees, etc.).
2. Check the listed package/packlet for an existing implementation.
3. Follow the link to read the actual exports/API before coding.
4. Add the package via `rush add -p <name>` (see `MONOREPO_GUIDELINES.md`).

If you cannot find what you need here, ask before adding a new dependency or rolling your own.

|---|---|
| [`base`](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-utils/src/packlets/base) | `Result<T>` (`succeed`/`fail`/`captureResult`), `mapResults`/`mapSuccess`/`mapFailures`/`allSucceed`, `MessageAggregator`, branded types (`Brand<T>`), normalization helpers, `generateUuid()` / `Uuid` / `isValidUuid` (cross-runtime UUIDv4 via `globalThis.crypto.randomUUID`). **Always use `Result<T>` for fallible ops.** |
| [`conversion`](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-utils/src/packlets/conversion) | `Converters.{string,number,boolean,object,arrayOf,enumeratedValue,optionalField,...}` for transforming/parsing untyped input into typed values. **Use instead of manual `typeof` checks + casts.** |
| [`validation`](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-utils/src/packlets/validation) | `Validators.{object,arrayOf,isA,oneOf,...}` for in-place validation of class instances or already-constructed objects. |
| [`collections`](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-utils/src/packlets/collections) | `ResultMap`, `ValidatingResultMap`, `Collector`, `ConvertingCollector`, `ValidatingCollector`, `KeyValueConverters`. **Result-friendly:** operations return `Result<T>` so they drop directly into `.onSuccess` chains. Use instead of raw `Map`/`Record` when keys/values need validation. `RetainingRingBuffer<T extends { seq: number }>` тАФ generic bounded most-recent-N ring with O(1) eviction, monotonic-`seq` cursor paging, and predicate filtering (`query({ sinceSeq?, limit?, filter? })`); `lastSeq` is stable across `clear()`. A **pure** storage substrate тАФ the caller mints each record (assigning its own `seq`/`timestamp`); the buffer only retains/evicts/queries. Use it as the shared mechanism beneath any "retain the most recent N records, page by cursor, filter" surface (observation stores, structured-log rings) instead of hand-rolling a circular buffer. |
| [`hash`](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-utils/src/packlets/hash) | `Crc32Normalizer`, `HashingNormalizer` тАФ deterministic hashing/normalization of arbitrary objects. `Normalizer.canonicalize(value)` тАФ RFC 8785 canonical JSON serialization: sorts object keys lexicographically (not numerically), emits directly to string (bypasses JS engine integer-key reordering). Available on `HashingNormalizer`, `Crc32Normalizer`, or any `Normalizer` subclass. Use for stable JSON fingerprinting, signing, or any protocol requiring byte-identical canonical form. |
| [`logging`](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-utils/src/packlets/logging) | `Logger`, `LogReporter` тАФ Result-friendly logger that can be inserted into a Result chain to log success/failure values without breaking the chain. `RetainingLogger` тАФ bounded most-recent-N ring of structured `ILogRecord` (seq + timestamp + level + formatted message + raw `args`) with a `getRecords({ minLevel?, sinceSeq?, limit? })` query API (min-severity filter, since-cursor incremental paging, most-recent-N tail); seq is monotonic and stable across eviction. `MultiLogger` тАФ fans every log call out to N child `ILogger`s, each applying its own threshold (e.g. feed a `ConsoleLogger` for stdout AND a `RetainingLogger` for an observability buffer from one pinned logger). Subclasses that need the structured form override the additive `LoggerBase._logStructured` hook (default no-op). |

| [`file-tree`](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-json-base/src/packlets/file-tree) | `FileTree`, `FsTree`, in-memory tree тАФ abstract directory/file traversal that works across Node/browser/zip/in-memory backends. **Default to `FileTree` for any file/directory access тАФ only fall back to raw `fs` if you have a concrete reason it cannot work.** |
| [`json-schema-builder`](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-json-base/src/packlets/json-schema-builder) (namespace `JsonSchema`) | Typed JSON Schema for the **LLM-tool subset** тАФ author the schema once, derive everything else. Seven factories (`JsonSchema.string`/`number`/`integer`/`boolean`/`enumOf`/`array`/`object`) plus the `optional` modifier return **`ISchemaValidator<T>`** instances that extend `Validator<T>` тАФ **the schema value IS the validator**, call `schema.validate(input)` directly. `JsonSchema.Static<typeof schema>` derives the TypeScript type automatically тАФ no caller-supplied `T`, no assertion. `schema.toJson()` emits the draft-07 wire schema (suitable for an LLM provider's tool `parameters` тАФ optionality becomes the parent's `required` array). `JsonSchema.fromJson(raw)` parses a raw `JsonObject` (e.g. an MCP-discovered schema) into an opaque `ISchemaValidator<JsonValue>` (runtime validation, no static `T`), failing with a JSON-pointer path on out-of-subset structural keywords (`$ref`, `oneOf`/`anyOf`/`allOf`, `not`, `if`/`then`/`else`, `pattern`, union `type`). Numeric validation is strict by default (rejects `'42'`; opt out with `number({ strict: false })`); objects are strict by default (`additionalProperties` defaults `false`, opt in with `{ additionalProperties: true }`). **Use this instead of authoring a JSON Schema and a separate Converter for the same shape тАФ the validator is derived from the schema, so they cannot drift.** Subset only: no `$ref`, polymorphism, `pattern`, custom formats, or JSON Schema 2020-12 features. |

### `@fgv/ts-prompt-assist` тАФ conditional prompt resolution
[libraries/ts-prompt-assist](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-prompt-assist)

ts-res-driven prompt authoring, organisation, and resolution for LLM workflows. Built on `@fgv/ts-res` (qualifier-conditional candidate selection), `@fgv/ts-extras` (`MustacheTemplate.create(template, { escape: 'none' })` for verbatim render; `AiAssist.fencedStringifiedJson` for fence-tolerant JSON output parsing; `Yaml.yamlConverter` for descriptor / bindings / qualifier-config loading), and `@fgv/ts-json-base` (`FileTree` for storage). **One YAML file per `(scope, prompt-id)`** holds the descriptor + all candidates; scope-level `_bindings.yaml` carries slot bindings; root-level `_qualifiers.yaml` carries ts-res `IQualifierDecl[]`. Key exports: **`PromptLibrary.create`** (factory; takes `store`, `qualifiers`, optional `registry` / `safetyPolicy` / `cacheListener` / `logger` / `observers`); **`PromptLibrary.resolve`** (chain-walks scopes, runs ts-res candidate selection on the supplied qualifier context, merges scope-level bindings cross-scope honouring `enforced` locks, recursively resolves resource bindings with RFC 8785 cycle-detection + depth cap, renders via Mustache with `escape: 'none'`, returns body + full `IPromptResolveTrace`); **`PromptLibrary.resolveJsonOutput<K>(req, rawOutput, expectedKind)`** (JSON pipeline strips fences тЖТ `JSON.parse` тЖТ typed Converter dispatched by `(id, kind)` тЖТ `outputValidations[]` chain narrowed by `value.kind`; runtime-verifies the descriptor's `output.kind === 'json'` AND that the descriptor's converter is registered to produce `expectedKind` before running the pipeline, so the return type `Extract<TResponse, { kind: K }>` is runtime-evidenced rather than caller-asserted); **`PromptLibrary.resolveFreeTextOutput(req, rawOutput)`** (runtime-verifies the descriptor's `output.kind === 'free-text'` and returns the raw output verbatim as `Result<string>` тАФ no `TResponse` involvement); **`PromptLibrary.describe`** (descriptor lookup across scopes with cross-scope structural-equality check). Storage: **`FileTreePromptStore`** (read-only at v0.1; works against `FsTree` / in-memory / zip / browser FSA File Tree); **`PromptStoreFixture.build`** (seeds an in-memory FileTree from authored records тАФ the canonical test/demo fixture; **no standalone `InMemoryPromptStore`**). Registry: **`PromptRegistry<TResponse>`** with three typed sub-registries (`converters` / `slotKinds` / `outputValidations`), parameterized by the consumer's response union (`TResponse extends { kind: string }`) so the JSON output pipeline flows the narrow Converter type end-to-end. Safety: **`IPromptSafetyPolicy`** carries `defaultMaxLength`, `screeners` (an ordered `IScreener[]` тАФ pluggable, async, per-slot-value pre-render screening; each `screen(ctx)` returns `Promise<Result<ISafeguardFinding[]>>`), and `antiJailbreakPreface` (post-render seam тАФ consumer supplies the framing). Screeners run sequentially in declaration order; a finding with `disposition: 'reject'` (or a screener returning `fail()`) fails the resolve and short-circuits the rest, while `'warn'` / `'info'` findings surface on `trace.safeguardFindings`. Findings carry open `kind`s (`SafeguardFindingKind = BuiltInFindingKind | (string & {})`), optional structured `metadata`, and `screener` attribution. **`createPatternScreener({ patterns, onMatch, screenedSources?, name? })`** is the built-in regex injection screener (reproduces the pre-pluggable semantics including `lastIndex` reset between values; `screenedSources` is the screener-owned source allowlist тАФ omit it to screen every sourced slot). Custom screeners (ML classifiers, remote calls) implement `IScreener` directly. The descriptor-level `safeguards.skipInjectionScreening` still bypasses all screeners for a prompt; `defaultMaxLength` and `antiJailbreakPreface` stay policy-level primitives (not screeners). Observability: additive **`IPromptLibraryCreateParams.observers?: ReadonlyArray<IPromptObserver>`** тАФ a fan-out fired once per public `resolve` / `resolveJsonOutput` / `resolveFreeTextOutput` call (never for nested resource-binding resolves; those roll up under the outer record's `trace.resourceBindingResolutions[].innerTrace`). Each **`IPromptObserver.observe(record): Promise<Result<unknown>>`** receives a `phase`-discriminated **`IPromptObservationRecord`** (`'resolve'` | `'json-output'` | `'free-text-output'`); output records cross-link to their resolve record via `linkedResolveSeq`; failure resolve records carry `error` only (no trace, v0.1). `seq`+`timestamp` are library-assigned (consistent across all observers); `contentHash` (CRC32 over canonical `{promptId,chain,qualifierContext,substitutions}`) is the dedupe key. **Observer errors never affect `resolve()`** тАФ a failed `Result`, throw, or rejection is swallowed and logged to the injected `logger` at warn; set **`fireAndForget: true`** on a slow remote observer so it does not extend `resolve()` latency (awaited is the default). Default store **`PromptObservationStore.create({ maxRecords?, qualifierResolver? })`** implements `IPromptObserver`, retains records in a bounded `RetainingRingBuffer`, and answers schema-aware **`query(criteria)`** (by `promptId` / `scope` / `qualifiers` (delegated to the injected resolver) / `outputKind` / `outcome` / `phase` / safeguard disposition / `seq` + `timestamp` range). **Qualifier matching is delegated to an injectable `IQualifierResolver`** тАФ the default `defaultStringEqualityQualifierResolver` does naive string equality on supplied (defined) axes, which is **deliberately narrower** than `PromptLibrary.resolve`'s ts-res-driven match (e.g. BCP47 `lang=fr-CA` records will NOT match a `lang=fr` query under the default). Deployments needing ts-res-equivalent semantics inject their own resolver; the swap is a pure implementation-of-interface change with no API surface churn on the store. **Most-permissive by design** тАФ it stores `body` / `rawOutput` / `substitutions` verbatim; redaction + retention are deployment policy, not library policy: wrap the store with a redacting `IPromptObserver` or substitute it entirely. **Do not roll your own prompt-observation ring** тАФ use `PromptObservationStore`, and for non-prompt retain-and-page surfaces use `RetainingRingBuffer` from `@fgv/ts-utils`. Branded ids: `PromptId` rejects `'::'`; `SlotName` matches the Mustache name production. **Use this for any LLM prompt that needs to vary on context (tenant, language, tone, region) or compose partial fragments over a shared base. Do not roll your own Mustache renderer for LLM prompts тАФ the verbatim render + double-brace rejection are load-bearing.** v0.1 is **out of scope** for: write API (`put` / `putBindings` / `delete`), change notification (`watch`), LLM-call orchestration, editor UX, default anti-jailbreak content, free-text output validators. See the [README](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-prompt-assist#readme) for the quick-start.

### `@fgv/ts-agent-memory` тАФ structured memory + knowledge substrate for agents
[libraries/ts-agent-memory](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-agent-memory)

App-agnostic **storage + retrieval substrate** for agent memory and knowledge тАФ a FileTree-backed markdown + YAML-frontmatter vault (git-trackable, cross-runtime via `@fgv/ts-json-base` `FileTree`). The consumer owns its curation / processing / composition; this package owns persistence, identity, typed bodies, attributed edges, dedup, write policy, retrieval, and the optional-layer seams. **Small invariant core + optional present-when-wired layers.**

**Core.** `IMemoryEnvelope` (branded `MemoryId` / `EntityId` / `Kind` / `Tag` / `LinkType` / `MemoryScopeKey`; store-owned transaction-time `created` / `updated` / `seq`; `contentHash`; structured extensible `IProvenance`; optional `temporal?` / `embeddingRef?`), `IEdge` (**attributed object edges** тАФ `type` / `target` / `confidence?` / `provenance?` / `valid_at?` / `invalid_at?` тАФ never bare strings), per-kind body via `IBodyConverterRegistry` (`register` / `registerSchema` / `has` / `getConverter` / `convert` тАФ consumer registers each kind's typed body Converter or `JsonSchema`; **no `any`**). `IIdentityCodec` maps a consumer domain key тЗД `{ scope, idStem }` (**no minted UUIDs тАФ `entityId` IS the domain address**); ships `KnowledgeIdentityCodec` (`<docId>`), `LtmIdentityCodec` (`<conversationId>`), `MtmIdentityCodec` (`<conversationId>:<turnIndex>` тЖТ `conversations/<id>/turn-<n>`).

**Store.** `FileTreeMemoryStore.create({ root, registry, codecs?, defaultCodec?, writePolicies?, scopeEncoding?, observers?, logger?, clock? })` тЖТ `IMemoryStore` (`get(kind, entityId)` codec-routed, `getById(scope, id)`, `list(filter)`, `put(record)`, `delete(kind, entityId)` тАФ all `Promise<Result<тАж>>`). Per write: body-validate тЖТ `Crc32Normalizer` content hash over `{ kind, body, links }` тЖТ **per-kind dedup** (`dedupScope: 'content'` = cross-id collapse for knowledge; `'entity'` default = same-id idempotent only, so distinct entities never collapse) тЖТ `IWritePolicy` admission (`KnowledgeLwwPolicy` last-write-wins; `MemoryCapCullPolicy` cap-cull-oldest over the `(scope, kind)` cohort + RFC-7386 **JSON Merge Patch** over `mutableFields`, composing `@fgv/ts-json` `JsonEditor` with `{ nullAsDelete: true, arrayMergeBehavior: 'replace' }`) тЖТ store-stamped `seq` / `created` / `updated` inside a write-lock тЖТ persist-then-evict (durability-first) тЖТ derived-index patch. Single-writer per store; the markdown files are the source of truth, the in-memory index is rebuilt on `create()`.

**Retrieval.** `IMemoryRetriever` + `IMemoryQuery` carrying `semantic?` / `asOf?` present-but-optional plus capability flags тАФ **stable against a future semantic / temporal backend with no resignature**. Ships `RecencyRetriever` / `TagRetriever` / `StructuredFilterRetriever` / `LinkTraversalRetriever` (bounded, cycle-safe BFS over the derived backlink index) + `HybridRetriever` (capability-union, injectable `IMergeStrategy`); `SemanticRetriever` **degrades loudly** (`Result.fail`) until an `IVectorIndex` is wired. **Observe.** `MemoryObservationStore` (ring-backed via `RetainingRingBuffer`; schema-aware `query`; the store fires `read` / `write` / `delete` observations when `observers` are wired, zero overhead otherwise).

**Shipped (v1):** the full substrate above (knowledge + experience kind families) **plus operational semantic recall** тАФ `InMemoryCosineIndex` (brute-force in-package cosine `IVectorIndex`) + embed-on-write (consumer-injected embedder; the core stays embedder-agnostic тАФ no `ai-assist` import). **Fast-follows (seams present, impl pending):** temporal versioned write path + temporal retrievers (envelope `temporal?` block + codec `isVersioned` flag already present); L2 agent-tool surface (`IAiClientTool`); L3 ingest orchestrator. **Use this for any agent memory / knowledge vault тАФ do not hand-roll a record store, content-hash dedup, or backlink index.** First consumer: PersonAIlity (knowledge-first, behind its `IKnowledgeSearchProvider`).

## Application shells (React)

### `@fgv/ts-app-shell` тАФ shared React app-shell primitives
[libraries/ts-app-shell](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-app-shell)

Reusable React shell components: top bar, sidebar, column cascade, modal, keyboard registry, responsive layout, theme, drop-zone, and the **`messages` packlet** (toast + collapsible log panel + observability bridge).

**Messages тАФ two-axis model.** A message separates the **filter axis** from the **display axis**:
- **`IMessage.level: MessageLogLevel`** (`'quiet'|'detail'|'info'|'warning'|'error'`, from `@fgv/ts-utils`) тАФ the canonical log level, and the **only** thing filtering keys off. The status-bar filter threshold is a `ReporterLogLevel` and the predicate is `Logging.shouldLog(message.level, threshold)`, so the panel filters at the same granularity a ts-utils logger records тАФ including `detail`. (`quiet` is only visible under the `all` threshold, per `shouldLog` semantics.)
- **`IMessage.severity?: MessageSeverity`** (`'info'|'success'|'warning'|'error'`) тАФ an optional **styling-only** override (color / icon / toast). When absent, styling is derived from the level via **`deriveSeverityFromLevel(level)`** (`quiet`/`detail`/`info`тЖТ`'info'`, `warning`тЖТ`'warning'`, `error`тЖТ`'error'`). `'success'` is a UI affordance with no log-level analog тАФ it lives only on this axis and is **never** derived; set it explicitly (e.g. a green "saved!" toast).

Create messages level-first: **`createMessage(level, text, options?)`** / **`addMessage(level, text, options?)`** where `options?: ICreateMessageOptions` carries `{ severity?, action? }`. The **`MessagesLogger`** / **`useLogReporter`** bridge implements the ts-utils `ILogger`/`LogReporter` interface and sets `level` straight through from the log call (lossless тАФ `RetainingLogger` levels map 1:1), leaving `severity` undefined so display styling is derived. `DEFAULT_TOAST_CONFIG` stays keyed by the styling `MessageSeverity`. **Import `MessageLogLevel`/`ReporterLogLevel`/`shouldLog` from `@fgv/ts-utils`** тАФ the messages packlet does not re-export them.

**Self-diagnosing setup.** `MessagesProvider` self-diagnoses missing Tailwind content-path setup via an in-panel warning message on mount тАФ a one-shot probe measures whether a sentinel arbitrary-value utility (`h-[7.3215px]`, chosen to be vanishingly unlikely to appear in any consumer's source so it can only be generated by Tailwind scanning ts-app-shell's built JS via the content glob) resolves to a sized element, and if it does not, appends a `'warning'` message with an `action.href` linking to the README setup section. Idempotent under React.StrictMode double-render and HMR; always-on (not dev-only). `IMessageAction` is a discriminated union (`IMessageHrefAction | IMessageCallbackAction`) тАФ exactly one of `href` (external link) or `onAction` (callback) is required at the type level.

**`StatusBar` layout props.** The collapsible log panel takes **`maxExpandedHeight?: string`** (CSS length, default `'40vh'`) тАФ the expanded panel is bounded to this height on both desktop and mobile, with the message list scrolling within the bound while the header/filter controls stay fixed; the mobile expanded view is an in-flow bounded sheet (with a dimming backdrop for tap-to-dismiss), not a `fixed inset-0` viewport takeover. **`defaultExpanded?: boolean`** (default `false`) controls expanded-on-mount. A discoverable collapse control (`тЬХ`) sits in the expanded header on both layouts.

A Result-integration boundary over [`@modelcontextprotocol/sdk`](https://github.com/modelcontextprotocol/typescript-sdk) (`^1.29.0`, a direct dependency) that connects to an MCP (Model Context Protocol) server, discovers its tools, and **adapts each into an `AiAssist.IAiClientTool`** (from `@fgv/ts-extras`) so it drops directly into `AiAssist.executeClientToolTurn` тАФ making any MCP server's tools callable across all four cloud providers with no per-provider work. All SDK imports are isolated to one file so the announced SDK v2 rename is a one-file change.

| Function | Return |
|---|---|
| `createStdioTransport({ command, args?, env?, cwd? })` | `Result<IMcpTransport>` |
| `createHttpTransport({ url, headers? })` | `Result<IMcpTransport>` |
| `connectMcpSession({ transport, clientName?, clientVersion?, logger? })` | `Promise<Result<IMcpSession>>` |
| `closeMcpSession(session)` | `Promise<Result<true>>` |
| `listMcpTools(session)` | `Promise<Result<ReadonlyArray<IMcpToolDescriptor>>>` (follows the SDK's `nextCursor` for the full catalog) |
| `callMcpTool(session, name, args)` | `Promise<Result<IMcpToolCallResult>>` (text-block projection; `isError: true` тЖТ `Result.fail`, never swallowed) |
| `adaptMcpTools(session, { logger? })` | `Promise<Result<{ tools: ReadonlyArray<AiAssist.IAiClientTool>; skipped: ReadonlyArray<IMcpSkippedTool> }>>` |

**Graceful degradation (load-bearing):** `adaptMcpTools` never fails the whole catalog over one bad schema. A tool whose `inputSchema` is outside the `JsonSchema.fromJson` subset (`$ref`/`oneOf`/`pattern`/union-`type`/тАж) is **excluded** from `tools` (the model is never offered a tool whose args we can't validate), **surfaced structurally** on `skipped` (name + JSON-pointer reason + raw failing schema), and тАФ when a `logger` is supplied тАФ **NOISY-warned** with all three. The `samples/testbed` `mcp-probe` scenario points it at any MCP server and prints a compatibility report (configure via `MCP_PROBE_URL` or `MCP_PROBE_COMMAND`).

**Explicitly NOT in scope:** browser sibling (`@fgv/ts-web-extras-mcp`), MCP resources / prompts / sampling, OAuth/managed auth (static headers only), multimodal tool-result passthrough, cross-server tool-name namespacing. The headline follow-on lever тАФ additively widening `JsonSchema.fromJson`'s subset (`$ref`/`$defs`, `pattern`) in `@fgv/ts-json-base` тАФ is a separate stream commissioned from what the probe surfaces. **Security:** `createStdioTransport` spawns a consumer-supplied command as a subprocess тАФ a trust boundary; never source the command from untrusted input.

### `@fgv/ts-extras-ollama` тАФ native Ollama sidecar Result boundary
[libraries/ts-extras-ollama](https://github.com/ErikFortune/fgv/tree/release/libraries/ts-extras-ollama)

**A Result-integration boundary over the official [`ollama`](https://github.com/ollama/ollama-js) JS library тАФ owning ONLY the native-API surface the OpenAI-compat `/v1` endpoint cannot express.** Node-only at v0.1. The text-completion / streaming / tool-use path is **NOT** here тАФ `@fgv/ts-extras/ai-assist` owns it via the `/v1` compat layer (point a provider descriptor's `endpoint` at `http://localhost:11434/v1` and call `callProviderCompletion` / `callProviderCompletionStream` / `executeClientToolTurn`). This package is the *native-only* complement: model management, streamed pulls, and grammar-constrained structured output.

`createOllamaClient({ host?, fetch?, headers? })` returns `Result<IOllamaClient>` (the opaque upstream `Ollama` handle, re-exported тАФ anything not wrapped here is reachable on it directly). Every primitive is client-first and returns `Promise<Result<T>>`:

| Primitive | Wraps | Returns |
|---|---|---|
| `listModels(client)` | `GET /api/tags` | `ReadonlyArray<IOllamaModel>` тАФ GGUF `size`/`family`/`parameterSize`/`quantizationLevel`/`modifiedAt` the `/v1/models` list can't give. |
| `listRunning(client)` | `GET /api/ps` | `ReadonlyArray<IOllamaRunningModel>` тАФ loaded models + `sizeVram` + `expiresAt`. |
| `showModel(client, model, { verbose? })` | `POST /api/show` | `IOllamaModelInfo` тАФ `modelfile`/`parameters`/`template`/`capabilities`/`modelInfo`. |
| `deleteModel(client, model)` | `DELETE /api/delete` | `IOllamaDeleteResult` (`{ model, deleted: true }` тАФ not `Result<void>`). |
| `pullModel(client, { model, insecure?, onProgress?, signal? })` | `POST /api/pull` (streamed) | `IOllamaPullResult` (`{ model, finalStatus, chunkCount }`). Drives the JSON-lines progress stream internally; `onProgress` fires per chunk; the terminal `Result` resolves when the stream ends. `AbortSignal` cancels the in-flight download. |
| `chatStructured<T>(client, { model, messages, schema, options?, keepAlive?, signal? })` | `POST /api/chat` with `format` | `IOllamaChatStructuredResult<T>` (`{ value, raw, model, doneReason? }`). |

**The headline win тАФ `chatStructured` no-drift, grammar-constrained output.** `schema` is a `JsonSchema.ISchemaValidator<T>` (`JsonSchema.object(...)` from `@fgv/ts-json-base`). The SAME object is BOTH the wire `format` (`schema.toJson()`, draft-07-sanitized to strip `$schema`/`additionalProperties` per the Gemini precedent) AND the reply validator (`schema.validate()`) тАФ they cannot drift. `T` is derived via `JsonSchema.Static<typeof schema>` тАФ no caller-supplied `T`, no cast. Ollama constrains the token sampler to the schema, so the reply is structurally guaranteed to match (a stronger guarantee than ai-assist's prompt-and-parse `generateJsonCompletion`, which asks-and-parses). `chatStructured` runs over the streaming chat path internally (the only path the `ollama` lib lets an `AbortSignal` cancel) and validates the assembled document whole.

**Dependency posture (mirrors `@fgv/ts-extras-transformers`):** `ollama` is a **peer + dev** dependency (bring your own pinned version); `@fgv/ts-utils` is **peer + dev** (consumer-provided, not installed transitively); only `@fgv/ts-json-base` is a hard **dependency** (`chatStructured` consumes `JsonSchema` as a first-class surface type).

**Explicitly NOT in scope:** text completion / chat / streaming (ai-assist via `/v1`); browser / CORS (`ollama/browser` + `OLLAMA_ORIGINS` тАФ a future `@fgv/ts-web-extras-ollama` sibling); model authoring (`push`/`create`/`copy`); `keep_alive`/lifecycle policy (pass-through only); pull-progress UI; multi-host orchestration / pooling / retries; native tool-calling on `/api/chat` (ai-assist owns tool turns). **Native embeddings (`embed`) are CUT** тАФ resolved by the `ai-assist-embeddings` design (OQ-1): Ollama embeddings are reachable via `/v1/embeddings` and are owned by `AiAssist.callProviderEmbedding` (the `ollama` descriptor's `baseUrl` already targets `http://localhost:11434/v1` тАФ pass the embedding model via `modelOverride`, and use the per-call `endpoint` override for a non-default host). Native `/api/embed` adds only marginal diagnostics (`total_duration`/`prompt_eval_count`) not worth a parallel path; revisit additively only if a concrete consumer needs them.

**Upstream:** `ollama` `^0.6.0` (peer dependency).

### Result-integration boundary тАФ package shape convention

When fgv wants to expose functionality from a well-maintained upstream library without baking in opinion or maintenance burden, the right shape is a thin **Result-integration boundary**: convert thrown exceptions / rejected promises into `Result<T>`, expose ~5тАУ8 primitive operations, and enumerate explicitly what is NOT in scope. No opinionated orchestration beyond the conversion.

Reference instances in this file:
- **`@fgv/ts-extras-webauthn` + `@fgv/ts-web-extras-webauthn`** тАФ six primitives wrapping `@simplewebauthn/*`; ceremony orchestration, challenge management, and credential storage are explicitly out of scope.
- **`@fgv/ts-extras-transformers` + `@fgv/ts-web-extras-transformers`** тАФ four primitives wrapping `@huggingface/transformers`; pipeline cache, model registry, device/quantization policy are explicitly out of scope.
- **`@fgv/ts-extras-mcp`** тАФ seven primitives wrapping `@modelcontextprotocol/sdk` plus one piece of glue (`adaptMcpTools` тЖТ `AiAssist.IAiClientTool[]`); browser sibling, MCP resources/prompts/sampling, OAuth, and multimodal passthrough are explicitly out of scope.
- **`@fgv/ts-extras-ollama`** тАФ six primitives (v0.1; native `embed` cut тАФ Ollama embeddings are owned by `AiAssist.callProviderEmbedding` via `/v1`) wrapping the `ollama` JS lib's *native* API (model management, streamed pull, grammar-constrained structured output); the completion path (ai-assist via `/v1`), browser/CORS, and model authoring are explicitly out of scope.

Both pairs have an explicit "NOT in scope" enumeration тАФ that list is load-bearing. A new integration-boundary package should carry the same shape: thin conversion + explicit not-in-scope list + no opinion added. See the WebAuthn pair's README as the prose template.

## Decision shortcuts (start here)

- **Error handling / fallible function?** тЖТ `Result<T>` from `@fgv/ts-utils/base`. Never throw across module boundaries.
- **Validating untyped JSON / config input?** тЖТ `Converters.object` from `@fgv/ts-utils/conversion` or `@fgv/ts-json-base/converters`. Never write `typeof x === 'object' && ... as T`.
- **Validating an existing class instance?** тЖТ `Validators` from `@fgv/ts-utils/validation`.
- **Map/collection with validation?** тЖТ `ResultMap` / `Collector` family from `@fgv/ts-utils/collections`.
- **Need a bounded most-recent-N ring of records with monotonic `seq` cursor paging, predicate filtering, and `lastSeq` preserved across `clear()`?** тЖТ `RetainingRingBuffer<T extends { seq: number }>` from `@fgv/ts-utils/collections`. A *pure* storage substrate тАФ the caller mints each record (assigning its own `seq` / `timestamp`); the buffer only retains / evicts / queries (`query({ sinceSeq?, limit?, filter? })`). Use it as the shared mechanism beneath any "retain the most recent N records, page by cursor, filter" surface (observation stores, structured-log rings) instead of hand-rolling a circular buffer.
- **Aggregating multiple errors?** тЖТ `MessageAggregator` + `mapResults` from `@fgv/ts-utils/base`.
- **Reading files (Node, browser, or zip)?** тЖТ `FileTree` from `@fgv/ts-json-base/file-tree` (+ `ts-extras/zip-file-tree` or `ts-web-extras/file-tree` for the backend).
- **Loading/saving JSON files?** тЖТ `JsonFsHelper` from `@fgv/ts-json-base/json-file`.
- **JSON templating or conditional inclusion?** тЖТ `JsonConverter` from `@fgv/ts-json/converters`.
- **Deep-merging JSON?** тЖТ `JsonEditor.mergeObjectInPlace` from `@fgv/ts-json/editor`.
- **Diffing JSON?** тЖТ `@fgv/ts-json/diff`.
- **Branded ID type?** тЖТ `Brand<T>` from `@fgv/ts-utils/base`.
- **Generating a UUID?** тЖТ `generateUuid()` from `@fgv/ts-utils/base` for the call-site one-liner; `ICryptoProvider.generateUuid()` from `@fgv/ts-extras/crypto-utils` when you already hold a provider or want a deterministic test override. Never call `globalThis.crypto.randomUUID()` directly.
- **Hashing / canonical hash of an object?** тЖТ `Crc32Normalizer` (`ts-utils/hash`) or `Md5Normalizer` (`ts-extras/hash` / `ts-web-extras/crypto-utils`).
- **Symmetric encryption / decryption (AES-GCM)?** тЖТ `DirectEncryptionProvider` or `KeyStore.encryptByName` from `@fgv/ts-extras/crypto-utils`, backed by `NodeCryptoProvider` (Node) or `BrowserCryptoProvider` (browser).
- **Need AES-256-GCM encrypt/decrypt with a key you already hold (PBKDF2/HKDF output, raw bytes)?** тЖТ `ICryptoProvider.encrypt(plaintext, key)` / `ICryptoProvider.decrypt(data, key, iv, authTag)` from `@fgv/ts-extras/crypto-utils`. The higher-level `DirectEncryptionProvider` and `KeyStore.encryptByName` paths wrap these for named-key lookup workflows.
- **Password-protected vault for keys / API keys / keypairs?** тЖТ `KeyStore` from `@fgv/ts-extras/crypto-utils`. Use `addSecretFromPassword` to derive + store, `verifySecretFromPassword` for constant-time password verification.
- **Need a private-key storage backend so `KeyStore.addKeyPair` works?** тЖТ **Node:** `CryptoUtils.KeyStore.EncryptedFilePrivateKeyStorage.create({ directory, encryptionKey, cryptoProvider })` тАФ import `CryptoUtils` from `@fgv/ts-extras` (or the `@fgv/ts-extras/crypto` subpath); one AES-256-GCM-encrypted JWK file per key via FileTree; you supply the 32-byte `encryptionKey`; `supportsNonExtractable: false`. **Browser:** `CryptoUtils.IdbPrivateKeyStorage.create({ databaseName?, storeName? })` тАФ import `CryptoUtils` from `@fgv/ts-web-extras` (IndexedDB; stores `CryptoKey` directly; `supportsNonExtractable: true`). Pass the instance as `KeyStore.create({ ..., privateKeyStorage })`. Don't roll your own `IPrivateKeyStorage`.
- **PBKDF2 key derivation, ECIES wrap/unwrap, asymmetric keypairs (Ed25519, ECDSA-P256, ECDH-P256, RSA-OAEP, X25519)?** тЖТ `ICryptoProvider` from `@fgv/ts-extras/crypto-utils`. Never call `node:crypto` or `crypto.subtle` directly.
- **Need to hash / verify a password (PBKDF2)?** тЖТ `KeyStore.addSecretFromPassword` / `verifySecretFromPassword` from `@fgv/ts-extras/crypto-utils`. Never roll PBKDF2 directly.
- **Need to hash / verify a password with Argon2id?** тЖТ `KeyStore.addSecretFromPasswordArgon2id` / `verifySecretFromPasswordArgon2id` from `@fgv/ts-extras/crypto-utils`, passing a `NodeArgon2Provider` (`@fgv/ts-extras-argon2`) or `BrowserArgon2Provider` (`@fgv/ts-web-extras-argon2`). Use `ARGON2ID_OWASP_MIN` or `ARGON2ID_PASSPHRASE` presets from `CryptoUtils` for OWASP-compliant parameters.
- **Need an Argon2id provider (Node)?** тЖТ `NodeArgon2Provider.create()` from `@fgv/ts-extras-argon2`.
- **Need an Argon2id provider (browser / WASM)?** тЖТ `BrowserArgon2Provider.create()` from `@fgv/ts-web-extras-argon2`.
- **Need stable canonical JSON (RFC 8785)?** тЖТ `Normalizer.canonicalize()` (available on `HashingNormalizer`, `Crc32Normalizer`, or any `Normalizer` subclass) from `@fgv/ts-utils` hash packlet.
- **Need to encode/decode a public key as multibase SPKI?** тЖТ `exportPublicKeyAsMultibaseSpki(key, provider)` / `importPublicKeyFromMultibaseSpki(encoded, algorithm, provider)` from `@fgv/ts-extras/crypto-utils`. Pass the `ICryptoProvider` instance so the underlying SPKI export/import routes through the correct platform implementation.
- **Need hybrid public-key encryption (encrypt to a recipient's public key, one-shot)?** тЖТ `HpkeProvider` from `@fgv/ts-extras/crypto-utils` (Node) or `@fgv/ts-web-extras` (browser). `HpkeProvider.create(subtle).orThrow()` then `sealBase(recipientPub, info, aad, plaintext)` / `openBase(recipientPriv, info, aad, enc, ciphertext)`. Suite: DHKEM(X25519, HKDF-SHA256) + HKDF-SHA256 + AES-256-GCM (RFC 9180 base mode). `info` is mandatory and must be application-specific context bytes тАФ empty `info` provides no context binding. Use `HpkeProvider.encodeEnvelope` / `decodeEnvelope` for wire transport. Never roll your own KEM.
- **Need standalone HKDF-SHA256 key derivation?** тЖТ `HpkeProvider.create(subtle).orThrow().hkdf(secret, salt, info, length)` from `@fgv/ts-extras/crypto-utils` (or `@fgv/ts-web-extras` for browser). Raw RFC 5869 HKDF тАФ unlabeled, suitable for application-level key derivation.
- **Need X25519 key agreement?** тЖТ `generateKeyPair('x25519', ...)` from `ICryptoProvider` (`@fgv/ts-extras/crypto-utils`) to produce the keypair; use `ICryptoProvider.exportPublicKeySpki` / `importPublicKeySpki` to exchange public keys. Raw `deriveBits` for the shared secret is not yet wrapped by `ICryptoProvider` тАФ do not call `crypto.subtle` directly.
- **Need to sign data or verify a signature (Ed25519, ECDSA-P256)?** тЖТ `ICryptoProvider.sign(privateKey, data)` / `ICryptoProvider.verify(publicKey, signature, data)` from `@fgv/ts-extras/crypto-utils`. Algorithm is inferred from the key's `algorithm` property тАФ generate keys with `generateKeyPair('ed25519', ...)` or `generateKeyPair('ecdsa-p256', ...)`. Both return `Promise<Result<...>>`.
- **Need HMAC-SHA256 authentication / MAC verification?** тЖТ `ICryptoProvider.hmacSha256(key, data)` / `ICryptoProvider.verifyHmacSha256(key, signature, data)` from `@fgv/ts-extras/crypto-utils`. The verify variant uses `timingSafeEqual` internally for constant-time comparison. Both return `Promise<Result<...>>`.
- **Need constant-time byte comparison?** тЖТ `ICryptoProvider.timingSafeEqual(a, b)` from `@fgv/ts-extras/crypto-utils`. Returns `boolean` directly (not `Result`). Node impl delegates to `crypto.timingSafeEqual`; browser impl uses XOR-walk accumulator. Never use `===` or loop with early return for security-sensitive comparisons.
- **Parsing a JSON-shaped string (with optional inner validation)?** тЖТ `Converters.stringifiedJson(inner?)` from `@fgv/ts-json-base/converters`.
- **Need typed JSON Schema for LLM tool authoring (one declaration тЖТ derived type + validator + wire schema)?** тЖТ `JsonSchema` from `@fgv/ts-json-base`. Author with `JsonSchema.object({ ... })` (factories: `string`/`number`/`integer`/`boolean`/`enumOf`/`array`/`object` + the `optional` modifier); `JsonSchema.Static<typeof schema>` derives the static type with no assertion; **the schema IS a Validator** тАФ call `schema.validate(input)` directly (no separate `toConverter` step); `schema.toJson()` gives the draft-07 wire schema for the provider's tool `parameters`; `JsonSchema.fromJson(raw)` parses an MCP-discovered `JsonObject` into an opaque `ISchemaValidator<JsonValue>` for runtime validation. **Don't author a schema and a separate Validator for the same shape тАФ they are the same object and cannot drift.** Strict numeric + strict object are the defaults (LLM convention). LLM-tool subset only (no `$ref`/`oneOf`/`pattern`/custom formats).
- **Asking an LLM for JSON and getting a validated `T`?** тЖТ `AiAssist.generateJsonCompletion<T>({ converter, ... })` from `@fgv/ts-extras/ai-assist`. Tolerates Markdown fences, preamble, and trailing prose by default. Don't reimplement fence-stripping in consumers.
- **Running a tool-use conversation with harness-side client tools?** тЖТ `AiAssist.executeClientToolTurn({ descriptor, apiKey, system?, messages, clientTools, tools?, model?, resolvedThinking? })` from `@fgv/ts-extras/ai-assist`. Like every turn entry point it takes the unified `{ system?, messages }` request shape (the last `messages` entry is the current user turn; preceding entries are history). Returns `{ events, nextTurn }` тАФ consumer drives the outer multi-turn loop. Pass `continuationMessages: continuation.messages` on subsequent turns тАФ `continuation.messages` is **cumulative** (replace each round, do not manually concatenate); do NOT put continuation wire objects in `messages` (they are a distinct post-current-turn axis). Works on **all four providers** (Anthropic, OpenAI Responses, Gemini, xAI via OpenAI). Type client tools as `IAiClientTool` (unparameterized) and validate args inside `execute` тАФ the callback is contravariant in `TParams`. Author the `parametersSchema` with `JsonSchema.object(...)` from `@fgv/ts-json-base`; `schema.validate(args)` is the round-trip verify step. **Gemini-specific:** cannot combine `web_search` (built-in grounding) with client tools in the same request тАФ Gemini's API HTTP-400s on that combination; pick one.
- **Making an MCP server's tools callable from an ai-assist tool-use conversation?** тЖТ `@fgv/ts-extras-mcp` (Node). `connectMcpSession` тЖТ `adaptMcpTools(session, { logger })` тЖТ hand `result.tools` (`AiAssist.IAiClientTool[]`) to `AiAssist.executeClientToolTurn({ ..., clientTools: result.tools })`. `adaptMcpTools` gracefully degrades: tools whose `inputSchema` is outside the `JsonSchema.fromJson` subset land in `result.skipped` (name + JSON-pointer reason + raw schema) and NOISY-warn rather than failing the catalog. Transports: `createStdioTransport({ command, args })` (spawns a subprocess тАФ trust boundary) or `createHttpTransport({ url, headers })`. Use the `samples/testbed` `mcp-probe` scenario (`MCP_PROBE_URL` / `MCP_PROBE_COMMAND`) to get a compatibility report for any server. **Don't hand-roll an MCP client or a JSON-SchemaтЖТclient-tool adapter.**
- **Diagnosing an OpenAI/xAI Responses stream that completes empty?** тЖТ Check `IAiStreamDone.incompleteReason?: string` on the `done` event. `'max_output_tokens'` тЖТ reasoning + tools consumed the output budget (raise the cap or shorten the prompt); `'content_filter'` тЖТ safety filter blocked the response. Absent тЖТ stream completed normally; if you still see no text, pass an `ILogger` to the streaming call (`callProviderCompletionStream({ ..., logger })` or `executeClientToolTurn({ ..., logger })`) and look for `ai-assist:unrecognized-event` warns тАФ the streaming adapter probably saw an event family it doesn't handle yet (provider-API evolution).
- **Alerting on provider-API evolution affecting `@fgv/ts-extras/ai-assist`?** тЖТ Pass an `ILogger` via the higher-level streaming params (`IProviderCompletionStreamParams.logger`, `IExecuteClientToolTurnParams.logger`, etc.) and filter on the stable `ai-assist:unrecognized-event` warn prefix. The streaming helpers thread the logger down to the per-adapter unknown-event sites; the per-adapter `IStreamApiConfig` itself is just `{ baseUrl, apiKey, model }` and is not where consumers wire the logger. The Anthropic and OpenAI Responses adapters maintain per-stream-dedup'd allowlists of recognized SSE event names; an unrecognized event triggers one warn per stream per name including a length-capped structural payload preview (top-level JSON keys + length by default; raw payload via the `AI_ASSIST_UNRECOGNIZED_EVENT_FULL_PAYLOAD` env var). Use this in production to alert when a provider's wire shape changes (likely class of issue: adapter parse miss тЖТ silent feature degradation). Gemini's drift instrumentation is queued separately (different wire shape).
- **Stripping LLM Markdown fences from raw text?** тЖТ `AiAssist.extractJsonText` (default extractor) or `AiAssist.fencedStringifiedJson({ extractor, inner })` (extension point) from `@fgv/ts-extras/ai-assist`.
- **Generating images from an LLM provider?** тЖТ `AiAssist.callProviderImageGeneration({ descriptor, apiKey, params })` from `@fgv/ts-extras/ai-assist`. Use `callProxiedImageGeneration(proxyUrl, ...)` to route through a proxy when direct browser calls are CORS-restricted. Pass top-level options (`size`, `quality`, `seed`, `count`) and a `models` array for provider-specific config blocks.
- **Declaring image generation capability on a provider?** тЖТ Add `imageGeneration: IAiImageModelCapability[]` to `IAiProviderDescriptor`. Each capability entry carries `format` (`'openai-images' | 'xai-images' | 'xai-images-edits' | 'gemini-image-out'`), `modelPrefix`, and optional flags like `acceptsImageReferenceInput`, `defaultOutputMimeType`, `outputParamStyle`, `supportsQualityParam`, `maxCount`.
- **Enabling extended thinking / reasoning on LLM completions?** тЖТ Pass `thinking: IThinkingConfig` to `callProviderCompletion` or `callProviderCompletionStream`. `IThinkingConfig` supports a cross-provider generic `effort?: 'low' | 'medium' | 'high'` shorthand and an optional `providers[]` array for per-provider config blocks. Provider-specific types: `IAnthropicThinkingConfig` (effort), `IOpenAiThinkingConfig` (effort, including `'none'` to re-enable temperature), `IGeminiThinkingConfig` (thinkingBudget integer), `IXAiThinkingConfig` (effort). Model-specific and `'other'` escape-hatch blocks let you override for individual models without affecting others. Merge precedence: generic effort тЖТ provider-generic blocks тЖТ model-specific/other blocks (later declaration wins within a tier). **Temperature + thinking conflicts**: Anthropic and OpenAI (non-`'none'` effort) and xAI return `Result.fail`; Gemini accepts both. Thinking content is discarded in this phase; caller-visible thinking-event surfacing is the scope of the followup stream `ai-assist-thinking-events`. Unknown/unsupported providers silently ignore the `thinking` field.
- **Managing a local Ollama sidecar тАФ listing / inspecting / pulling / deleting models, or grammar-constrained structured output?** тЖТ `@fgv/ts-extras-ollama` (Node-only). `createOllamaClient({ host? })` тЖТ `Result<IOllamaClient>`, then `listModels` / `listRunning` / `showModel` / `deleteModel` (model management the `/v1` layer can't express), `pullModel({ model, onProgress?, signal? })` (streamed download progress тЖТ terminal `Result`), and **`chatStructured<T>({ model, messages, schema, signal? })`** for grammar-constrained JSON: the `JsonSchema.object(...)` schema is BOTH the wire `format` (draft-07-sanitized) AND the `schema.validate()` reply validator тАФ one declaration, no drift, `T` derived via `JsonSchema.Static`. **For text completion / streaming / tool-use against the same daemon, this is the WRONG package тАФ use `@fgv/ts-extras/ai-assist` with a provider `endpoint` of `http://localhost:11434/v1`.** Native `embed` is **CUT** тАФ Ollama embeddings are owned by `AiAssist.callProviderEmbedding` via `/v1/embeddings` (see the "text тЖТ vector" decision shortcut and OQ-1), not this package. `ollama` is a peer dependency.

- **Need a text embedding (`text тЖТ vector`)?** Three paths; pick by where the weights run:
  - **In-process / local / offline (you own the model lifecycle)** тАФ on-device RAG pre-filter, privacy-sensitive, zero per-call cost тЖТ `embed` / `loadPipeline('feature-extraction', тАж)` from **`@fgv/ts-extras-transformers`** (Node) or **`@fgv/ts-web-extras-transformers`** (browser). Returns a raw `Tensor` (pass `{ pooling: 'mean', normalize: true }` for a sentence vector). You manage model download / cache / device / quantization.
  - **Cross-provider cloud HTTP** тАФ OpenAI `text-embedding-3-*`, Gemini `gemini-embedding-001`, Mistral `mistral-embed`, **or a self-hosted OpenAI-compatible / Ollama server via the `endpoint` override** тЖТ `AiAssist.callProviderEmbedding` from **`@fgv/ts-extras/ai-assist`**. Batch in, `number[][]` out, `Result`-wrapped, descriptor-driven. **This is also the Ollama answer** тАФ point it at `http://localhost:11434/v1`; there is no separate Ollama-native embedding API (the `@fgv/ts-extras-ollama` native `embed` was cut тАФ see OQ-1).
  - One-line mental model: **`@fgv/ts-extras-transformers` = the weights run in *your* process; `callProviderEmbedding` = the weights run on *a server you `fetch`*.** Same local-vs-distant split as completion.
- **Embedding text via a cloud or self-hosted provider (OpenAI / Gemini / Ollama / openai-compat / Mistral)?** тЖТ `AiAssist.callProviderEmbedding({ descriptor, apiKey, params: { input, dimensions?, taskType? }, modelOverride?, endpoint? })` from `@fgv/ts-extras/ai-assist`; `callProxiedEmbedding(proxyUrl, ...)` to route through a backend proxy. `input` is `string | string[]`; the result's `vectors[i]` aligns to `input[i]`. Pass `dimensions` (honored by OpenAI `text-embedding-3-*` and Gemini) and/or `taskType` (Gemini retrieval asymmetry, e.g. `'retrieval-query'` vs `'retrieval-document'`) тАФ both are a **logged no-op** on providers/models that don't support them, never a failure, so a cross-provider call site passes them once. Self-hosted (`'ollama'`/`'openai-compat'`) needs an explicit `modelOverride` (no default embedding model). xAI / Anthropic have no embeddings endpoint and reject up front. **Don't reach for `@fgv/ts-extras-transformers` for this** тАФ that's the in-process path; this is the distant-HTTP path (see the three-way decision shortcut above).

The `'ollama'` descriptor defaults to `http://localhost:11434/v1`; `'openai-compat'` has `baseUrl: ''` and requires an explicit `endpoint`. **`defaultModel` is `''` for both** тАФ always supply `model` explicitly (e.g. `model: 'llama3.2'`). - **Authoring, versioning, or resolving LLM prompts that need to vary on context (tenant / language / tone / region) or compose partial fragments?** тЖТ `PromptLibrary.create` from `@fgv/ts-prompt-assist`. One YAML file per `(scope, prompt-id)` carries the descriptor + ts-res-conditional candidates; scope-level `_bindings.yaml` supplies slot bindings; root-level `_qualifiers.yaml` (optional) carries the qualifier config. `PromptLibrary.resolve` chain-walks scopes, runs ts-res candidate selection, merges bindings, renders via Mustache with `escape: 'none'`, and returns body + full `IPromptResolveTrace`. **Do not roll your own LLM-prompt loader / renderer; the verbatim Mustache + double-brace rejection are load-bearing.**
- **Binding a prompt slot to the body of another prompt?** тЖТ Author a `SlotBinding` with `kind: 'resource', resourceId: '<inner-prompt-id>'` in `_bindings.yaml` or as the slot's `defaultBinding`. `PromptLibrary` resolves the inner prompt recursively (depth-capped via `IPromptLibraryCreateParams.resourceBindingDepthLimit`, default 5; cycle-detected via RFC 8785 canonical-JSON keys) and substitutes the inner body into the outer slot. The full inner trace surfaces in `trace.resourceBindingResolutions[].innerTrace`.
- **Composing one prompt from several resolved prompts (horizontal / multi-contributor composition)?** тЖТ Use **`HorizontalComposer`** from `@fgv/ts-prompt-assist` тАФ the safety-closed, first-class composer. Resolve each contributor independently via `PromptLibrary.resolve`, then `HorizontalComposer.create({ contributors, logicalSlots, composedDescriptor, composedBody, safetyPolicy? }).onSuccess((c) => c.compose())`. Each **`IContributorSpec`** pairs a resolved prompt with a composer-layer **`provenance: number`** (provenance is orthogonal to ts-res qualifier selection тАФ contributors are already fully resolved). **`ILogicalSlotConfig[]`** (code-authored at v0.1 тАФ no YAML converter yet) declares the composed prompt's logical-slot space: each carries a `logicalSlotName`, an **explicit `contributorSlots` map** (`{ contributorProvenance, slotName }[]` тАФ name-matching is never implicit; every reference is checked at `create()` against the contributor's `IResolvedPrompt.slots`), a `strategy` (**`LogicalSlotStrategy = 'concatenate' | 'overwrite'`**), and an optional `separator` (default `'\n\n'`). **Directive-aware merge (load-bearing):** per logical slot, contributions are ordered by **ascending provenance** (declared `contributorSlots` order is a stable tiebreaker); **`constraint`-directive contributions are always concatenated first and NEVER dropped, regardless of strategy**; non-constraints then apply the declared strategy on top (`concatenate` joins all ascending with the separator; `overwrite` keeps only the highest-provenance non-constraint). `create()` does synchronous build-time map validation (unique provenance; every referenced contributor + slot exists). `compose()` runs the **directive-aware merge тЖТ `applySafeguards` over the merged slot map against the `composedDescriptor`** (per-logical-slot `maxLength` / `allowedDirectives`; reject тЖТ `Result.fail`, warn/info тЖТ `IComposedPrompt.safeguardFindings`) **тЖТ Mustache render** of `composedBody` (`escape: 'none'`). Returns **`IComposedPrompt`** (`body` + `mergedSlots: ReadonlyMap<SlotName, IResolvedPromptSlot>` + `provenanceTrace: ReadonlyMap<SlotName, ISlotProvenanceEntry[]>` + `safeguardFindings`). This closes the interim safety gap of the consumer-side external-composer path (which reads `IResolvedPrompt.slots` directly and must self-screen). **`IResolvedPrompt.slots`** remains the supported per-slot read surface for callers who want the resolved values without the full composer; the logical-slot processing order is **not** semantically binding (declared order is a tiebreaker for topo-equal slots тАФ a future dependency topo-sort phase may reorder without a breaking change). **Do not hand-roll an external composer or concatenate rendered bodies тАФ use `HorizontalComposer`.**
- **Observing every prompt resolution as it flows (audit / debug / "where did this body come from" / per-tenant traces / offline analysis)?** тЖТ Pass `observers?: ReadonlyArray<IPromptObserver>` to `PromptLibrary.create({ ..., observers })` from `@fgv/ts-prompt-assist`. The library fans out at the three public boundaries (`resolve` / `resolveJsonOutput` / `resolveFreeTextOutput`); never fires for nested resource-binding resolves (those roll up under the outer record's `trace.resourceBindingResolutions[].innerTrace`). For the default in-memory store, use **`PromptObservationStore.create({ maxRecords?, qualifierResolver? })`** тАФ implements `IPromptObserver` and exposes a schema-aware `query(criteria)` (by `promptId` / `scope` / `qualifiers` / `outputKind` / `outcome` / `phase` / safeguard disposition / `seq` + `timestamp` range). For high-latency observers (writing to SIEM / remote store), set `fireAndForget: true` on the observer to dispatch without extending `resolve()` latency; default is awaited so the record lands before return. **Observer errors never affect `resolve()`** тАФ failed `Result`, throw, or rejection is swallowed and logged to the injected `logger` at warn. **The default store is most-permissive** (stores `body` / `rawOutput` / `substitutions` verbatim) тАФ redaction + retention are deployment policy: wrap the store with a redacting `IPromptObserver` or substitute it entirely. `seq` + `timestamp` are library-assigned (consistent across every observer); output records cross-link to their resolve record via `linkedResolveSeq`. **`qualifiers` matching is delegated to an injectable `IQualifierResolver`** тАФ default is string equality on supplied axes (deliberately narrower than `PromptLibrary.resolve`'s ts-res match); inject `IQualifierResolver` for similarity-aware semantics. **Do not roll your own prompt-observation ring** тАФ use `PromptObservationStore`, and for non-prompt retain-and-page surfaces use `RetainingRingBuffer` from `@fgv/ts-utils`.
- **Validating a JSON-shaped LLM response against a typed `Converter<T>` + validator chain?** тЖТ `PromptLibrary.resolveJsonOutput<K>(req, rawOutput, expectedKind)` from `@fgv/ts-prompt-assist`. Build a `PromptRegistry<TResponse>` where `TResponse` is the consumer's response union (each member carries a `kind: '<discriminator>'` field). Register Converters via `registry.converters.register(id, kind, converter)` and validators via `registry.outputValidations.register(id, { appliesTo, validate })`. Caller passes `expectedKind` as a string literal (`'cited'`, `'classifier'`, etc.); `K` is inferred and the return type narrows to `Extract<TResponse, { kind: K }>` automatically. At the public-API entry, `resolveJsonOutput` runtime-verifies that the descriptor's `output.kind === 'json'` AND that `descriptor.output.converterId` is registered to produce `expectedKind` тАФ a mismatch (caller asked for `'cited'` but the descriptor's converter produces `'classifier'`) rejects loudly with the prompt id + actual-vs-expected kinds, *before* the pipeline runs. After the entry check passes, the pipeline strips Markdown fences (via `AiAssist.fencedStringifiedJson`), `JSON.parse`s, dispatches to the Converter recorded under `(id, kind)` (no cast between the Converter type and validator inputs), then chains the validators (each narrowed by `value.kind тИИ appliesTo`). Loader-side checks reject incompatible descriptors at `describe()` / `resolve()` time (belt); the runtime path re-checks `value.kind` per validator (suspenders). The combined entry-check + belt + suspenders pin every degree of caller-asserted-`T` freedom that the previous `resolveAndValidateOutput<T>` API exposed тАФ there is no longer a path where the return type's runtime kind diverges from what the descriptor declared. **Free-text output:** for descriptors whose `output.kind` is `'free-text'`, use `resolveFreeTextOutput(req, rawOutput)` тАФ it runtime-verifies the descriptor's output kind and returns the raw output verbatim as `Result<string>`. No `TResponse` involvement; no Converter; no caller assertion.
- **Enforcing safety policies (length cap, pluggable value screeners, anti-jailbreak preface) on a resolved prompt?** тЖТ `IPromptLibraryCreateParams.safetyPolicy` from `@fgv/ts-prompt-assist`. Carries `defaultMaxLength`, `screeners` (ordered `IScreener[]`; each `screen(ctx)` is async and returns `Result<ISafeguardFinding[]>`), and `antiJailbreakPreface` (consumer-supplied post-render framing тАФ library ships no default content). Use `createPatternScreener({ patterns, onMatch, screenedSources? })` for regex injection screening (`lastIndex` reset between values; `screenedSources` is the screener-owned source allowlist). Screeners run sequentially; the first `disposition: 'reject'` finding (or a screener `fail()`) fails the resolve and short-circuits the rest. Warn / info findings (screener findings under `'warn'` / `'info'`, plus `'enforced-override-ignored'` from the merge) surface in `trace.safeguardFindings`; reject paths (length-cap violations, reject findings, screener `fail()`s) fail the resolve with the rejection in the error message rather than on a trace. Findings carry open `kind`s, optional `metadata`, and `screener` attribution. Implement `IScreener` directly for ML-classifier or remote-call screeners.
